Post-Merge Readiness Review

The Box · branch claude/review-box-app-blockers-1einj · 2026-05-11

Verdict: not yet ready, but materially closer. Eight of the eleven pre-merge blockers verify clean and stay clean. Five blocker-level issues remain or were newly surfaced — one is the deferred submitGuess transaction (B5), and four are issues the first pass missed or under-prioritised (image-proxy enumeration, first-call timer fallback, Knex SSL, first-user admin race). All five are small fixes. Estimate: 1–2 engineer-days plus one E2E pass.

Method

Same four parallel audit lanes as the original review — security, deploy / infra, data integrity, realtime / queues — rerun against HEAD after the PR #206 squash-merge (commit 06d683c). Each lane was asked to verify the prior fixes (PASS / PARTIAL / FAIL) before opening new ground. Where two lanes disagreed about a finding, the conflict was resolved by reading the cited code directly — one agent's claim that Math.max(server, client) is exploitable for a 200× multiplier is wrong (slower time produces a lower multiplier, so the picked direction is correct); the residual risk in B4 is the silent fallback when round-stamping fails, which is a real but separate issue.

Open blockers

B5 (open) — submitGuess is still not transactional Blocker

packages/backend/src/domain/services/game.service.ts ~440–750

The earlier deferral note assumed the worst case was “score loss, not score gain.” That's correct for the tier-session score itself (the final UPDATE is overwrite-style, so a concurrent winner wipes the loser), but two adjacent bugs slip through anyway:

  • Wrong-guess and correct-guess counters can diverge from guesses. The service reads getCorrectAnswersCount and then writes correct_answers + (isCorrect ? 1 : 0) on top of a stale value, so two concurrent correct guesses can end with a counter that doesn't match the actual row count in guesses.
  • Second-chance floor can be applied to two guesses rows. The activation row is correctly marked once (the whereNull('applied_to_guess_id') guard holds), but both threads independently raised their in-memory scoreEarned to the floor before saving their guess row. Anything that later reads guesses.score_earned directly (achievements, audit reports, future score replays) sees the boost twice.
Fix

Wrap the route handler body in db.transaction(async (trx) => ...), thread trx through saveGuess, updateTierSession, updateGameSession, useItems, and markApplied. Add a unique index on guesses(tier_session_id, position) WHERE is_correct = true if you want duplicate-correct submits to be a hard error rather than just wasted work.

New-1 — Image proxy lets any logged-in user enumerate every screenshot Blocker

packages/backend/src/presentation/routes/game.routes.ts:105–143

GET /api/game/image/:screenshotId is gated by authMiddleware but does not check that the requested screenshot belongs to a challenge the user is currently playing or has already completed. A logged-in user (free or anonymous session) can iterate IDs from 1 upward and pull every screenshot ever uploaded, including ones scheduled for future daily challenges. Combined with H3 below, the answer leak is essentially “authenticate once, harvest forever.”

Additionally, the response hard-codes Content-Type: image/jpeg with no X-Content-Type-Options: nosniff header and no extension check on the stored path. If an admin import ever lands an SVG, the browser will render it with embedded JavaScript on the same origin as the Better Auth session cookie.

Fix

Resolve the screenshot to its (tier, daily_challenge) and verify the user has a game_session for that challenge (completed or in-progress). Add X-Content-Type-Options: nosniff to the response and reject non-JPEG/PNG extensions before res.sendFile.

New-2 — B4 round-stamping fails silently and falls back to the client value Blocker

packages/backend/src/domain/services/game.service.ts:402–418 (getScreenshot)

The merged fix wraps markRoundStarted in a try/catch that logs a warning and continues, with the rationale that “submitGuess will fall back to the client-supplied timer.” That fallback is the exact behaviour B4 was trying to remove. A flaky DB write, a stalled connection in the pool, or even a deliberately-triggered error (e.g., interleaving a rejected request that holds a row lock) leaves round_started_at = NULL and re-enables the original { roundTimeTakenMs: 1 } cheat for that round.

Note: the “clock-skew exploit” one of the re-audit agents raised against this fix is incorrect — Math.max(server, client) picks the slower time, which produces the lower multiplier, not the higher one. The real residual is the fallback path, not the comparison direction.

Fix

Treat round-stamp failure as a 500 (or retry once and then 500). Better still: persist round_started_at at session start with a default of started_at, so the column is never null on a live row, then reject any submitGuess whose tier session has a null/stale round_position.

New-3 — Knex production pool has no SSL for remote Postgres Blocker

packages/backend/knexfile.ts

Was H5 in the original review and not in scope of PR #206. Worth re-flagging as a launch blocker because every managed Postgres provider (RDS, Supabase, Neon, DigitalOcean) requires TLS on connection. Today the production block sets only min/max; ssl is undefined, so the pg driver opens an unencrypted socket and the DB rejects it. Locally the bug is invisible because the dev DB doesn't enforce TLS.

Fix 
production: {
  client: 'pg',
  connection: {
    connectionString: env.DATABASE_URL,
    ssl: env.DATABASE_URL.includes('localhost')
      ? false
      : { rejectUnauthorized: false },
  },
  pool: { min: 2, max: 10, idleTimeoutMillis: 30_000 },
}

New-4 — First-user admin bootstrap race (re-flagged) Blocker

packages/backend/src/infrastructure/auth/auth.ts:237–249

Was H4 in the original review and not in scope of PR #206. The before Better Auth hook reads SELECT COUNT(*) FROM "user" and then returns the inserted user with role: 'admin' if zero. Two parallel sign-ups on a fresh DB both read zero and both become admin. Low probability on a quiet launch night, high blast radius on launch day if the link is shared widely.

Fix

Either enforce a partial unique index (CREATE UNIQUE INDEX one_admin ON "user"(role) WHERE role = 'admin') and let the second writer collide, or move the check + insert into a serialisable transaction.

High severity

H-1 — Wrong-guess response still includes future hints High

game.service.ts:825–830 (and the completion path at 780–789)

H2 from the prior review. Still present: an incorrect guess returns availableHints with the correct game's year, publisher, developer, and genre. A player can spend one bad guess to harvest everything they need for the next attempt.

H-2 — Correct gameId with empty text still wins High

game.service.ts:464–465

The shortcut data.gameId === screenshot.gameId bypasses the fuzzy matcher entirely, so a client only needs to know the numeric game id of the answer — not the name. Combined with the image-proxy enumeration in New-1, an attacker who has scraped the id→file mapping wins every screenshot with { gameId: N, guessText: '' }.

Fix

Require either a non-empty guessText that fuzzy-matches, or move correctness scoring entirely to the text path and treat gameId as a UX hint for the autocomplete only.

H-3 — Docker healthcheck probes liveness, not readiness High

Dockerfile:125–126 vs /healthz in index.ts:199

The container probe hits /health, which is a static-200 liveness route. The DB- and Redis-touching route is /healthz. A backend stuck behind a dead Postgres still reports healthy, so Traefik will keep routing traffic to it. Switch the HEALTHCHECK to /healthz.

H-4 — Request logger leaks tokens in URLs High

packages/backend/src/presentation/middleware/request-logger.middleware.ts:11

Was H8 in the prior review. Every request logs req.url verbatim, so password-reset and email-verification tokens (the short-lived ones that are the credential) end up in log aggregators. Switch to req.path and never serialise query strings or bodies from /api/auth/*.

H-5 — express.json() and Stripe raw parser have no explicit limit High

backend/src/index.ts:86-87 and billing-webhook.routes.ts:85

Default express.json() caps at 100 kb, which is fine, but pinning it makes future refactors safe. More importantly, the Stripe webhook uses express.raw({ type: 'application/json' }) with no limit, and any caller who reaches that route (it's signature-checked, but the buffer is allocated before verification) can balloon memory before the signature fails.

Fix 
app.use(express.json({ limit: '256kb' }))
app.use('/api/billing/webhook', express.raw({ type: 'application/json', limit: '1mb' }), ...)

H-6 — Leaderboard ordering has no deterministic tie-breaker High

leaderboard.repository.ts:24–52

Same-score players swap ranks on every refresh because the only ORDER BY clause is total_score DESC. Add ORDER BY total_score DESC, completed_at ASC, user_id ASC and switch the displayed rank to a window function (DENSE_RANK() OVER (ORDER BY total_score DESC)) so ties don't push everyone else down a slot.

H-7 — Backup credentials passed via PGPASSWORD env High

compose.yml:115-116 + scripts/db-backup.sh

Was M1 in the prior review; the infra re-audit re-elevated it. Whether to call this a blocker or a high depends on the team's threat model for a self-hosted Docker host. Either way it should move to Docker secrets or a 0600 /run/secrets/db_password file before launch.

Medium severity

M-1 — Catch-up sessions can be replayed if abandoned mid-game Medium

game.service.ts:345–366

startChallenge blocks a re-run only when is_completed = true. An abandoned catch-up session can be resumed indefinitely, and the leaderboard filter assumes catch-up plays are filtered everywhere. Tighten the guard to (is_catch_up = false) OR (is_completed = false) so a completed catch-up can't be re-entered even if some other code path flips is_completed = false.

M-2 — Streak-freeze auto-consume race (known) Medium

daily-login.service.ts:200–240

Still present, still in-code commented as “acceptable for v1.” Worth moving into the same transaction as the streak update before public launch — a single double-decrement on a high-traffic day is enough to generate a support ticket.

M-3 — Recurring-job re-registration race on parallel boot Medium

index.ts:364–387

Still present. BullMQ's jobId dedup keeps it from producing duplicates, but a rolling deploy with overlap can leave a short window where the queue has zero pending instances of a recurring job. For a daily-challenge job that fires once at 00:00 UTC the window matters; for the rest it's noise.

M-4 — Shutdown does not stop active workers before closing the queue Medium

index.ts:703–744

The graceful-shutdown chain calls worker.close() but not worker.getWorker().stop() first, so a job that's in flight at SIGTERM can keep running for up to the 8 s timeout window before its connection is yanked. Mid-write rows in tier_sessions or guesses are the obvious casualty.

M-5 — VITE_USE_MOCK_API has no build-time assertion Medium

packages/frontend/vite.config.ts + services/*

The runtime guard is correct (=== 'true' string comparison, defaults to undefined). It just doesn't fail loud if a production build is accidentally produced with the flag set. A one-line check in vite.config.ts that throws when mode === 'production' && process.env.VITE_USE_MOCK_API === 'true' closes the foot-gun.

False positives surfaced by the re-audit

Flagged by one of the re-audit lanes, dismissed on second read:

  • “Math.max(server, client) lets an attacker score 200 pts via a 10 s timer.” No — calculateSpeedMultiplier at game.service.ts:55-68 gives 2.0× only for <3 s; 10 s yields 1.0× = 100 pts. Math.max is the correct direction. The B4 residual is the silent NULL fallback (New-2), not the comparison.
  • “Second-chance floor doubles the user's score by 70+70.” The tier-session score update is overwrite, not increment — the second write wipes the first. The real bug is that two guesses rows record the floored value, which is the case New-1 / B5 cite.
  • “CSRF protection is missing.” Better Auth sets cookies with SameSite=Lax by default, which blocks cross-site form POSTs to the API. Combined with the same-origin CORS configuration, there's no realistic CSRF surface today. Could be tightened to Strict, but it's not a launch blocker.
  • “Avatar deletion is path-traversal vulnerable.” Only via a second, separately-required bug that gives the attacker write access to user.avatarUrl. Defense-in-depth is welcome, but it's not exploitable today.

Pre-launch checklist (delta on top of PR #206)

  • B5 — submitGuess wrapped in a Knex transaction; unique partial index added on guesses(tier_session_id, position) WHERE is_correct.
  • New-1 — image proxy enforces session ownership; X-Content-Type-Options: nosniff set; extension allowlist enforced.
  • New-2 — B4 round-stamp failure is fail-closed; column gets a non-null default on session start.
  • New-3 — Knex ssl set for any non-localhost DATABASE_URL.
  • New-4 — first-user admin race closed by partial unique index or serialisable transaction.
  • H-1, H-2 — hints suppressed on wrong guess; correctness requires non-empty text.
  • H-3 — Dockerfile HEALTHCHECK switched to /healthz.
  • H-4 — request logger uses req.path; auth routes opt out of body/query serialisation.
  • H-5 — explicit limit on express.json() and on the Stripe express.raw().
  • H-6 — leaderboard ORDER BY has stable tiebreakers; ranks via window function.
  • H-7 — DB-backup credentials moved off PGPASSWORD.
  • One full npm run build && npm run lint && npm test && npm run test:e2e pass on the fix branch.
  • Manual smoke: two-tab guess race, ten back-to-back catch-up resumes, one rolling deploy under load to validate SIGTERM behaviour.
