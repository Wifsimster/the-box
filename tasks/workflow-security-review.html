GitHub Actions — workflow security review

Scope: .github/workflows/ci.yml, release.yml, deploy.yml, pages.yml on main @ 5682a88 (2026-06-09). Repository is public.

internal · senior developers only

Verdict

The permissions model is fundamentally sound: default contents: read everywhere, write scopes elevated only on the single job that needs them, no pull_request_target, no untrusted event data (PR titles, issue bodies, branch names) interpolated into shell. The findings below are about the operating environment the workflows run in, not the workflow logic itself — the worst one comes from combining a public repository with a self-hosted homelab runner.

F1 · Self-hosted runner on a public repository High

deploy.yml:25 — runs-on: [self-hosted]

deploy.yml itself is triggered safely (workflow_run gated on Release success on main, plus manual dispatch). The problem is the runner, not this workflow: on a public repo, a fork PR ships its own modified workflow files, and any workflow with an unrestricted pull_request trigger (like ci.yml) can be edited in the fork to declare runs-on: [self-hosted]. If that run is allowed to start, arbitrary attacker code executes on the homelab box — which has the Docker socket (root-equivalent), the production compose stack in /opt/docker/the-box with its .env secrets, and LAN access. GitHub's own docs recommend never attaching self-hosted runners to public repositories.

The default protection ("require approval for first-time contributors") is weak here: any account with one previously merged PR — even a typo fix — runs workflows without approval forever after.

  • Minimum, do now: Settings → Actions → General → Fork pull request workflows → "Require approval for all outside collaborators".
  • Better: make the runner ephemeral (registered with --ephemeral, re-provisioned per job, e.g. via a container or VM snapshot) so a compromise doesn't persist, and drop its Docker-socket access in favor of a constrained deploy user.
  • Structural alternative: invert the flow — keep deploys pull-based on the homelab (systemd timer or webhook listener that verifies a signature) so no GitHub-controlled execution ever lands on the box. This restores the Watchtower trust model while keeping the observable log.

F2 · npm ci with a persisted write token in publish Medium

release.yml:240–258 — checkout persists GITHUB_TOKEN (contents: write), then npm ci

actions/checkout stores the job's GITHUB_TOKEN in .git/config by default. In the publish job that token has contents: write (it later pushes the version bump and tag to main). npm ci then executes every dependency's lifecycle scripts — a compromised transitive dependency (the npm-supply-chain scenario) can read that token and push to main, tag releases, or tamper with the repo. The checks and decide jobs only hold read tokens, so the blast radius is specific to publish.

  • Run npm ci --ignore-scripts in publish — the version bump (npm run version:* → npm version in workspaces) doesn't need dependency lifecycle scripts.
  • Add persist-credentials: false to the checkout and authenticate explicitly only for the push step (e.g. git push https://x-access-token:${GITHUB_TOKEN}@github.com/… with the token passed as env to that single step).

F3 · Production logs in public Actions logs Medium

deploy.yml:60–68 — docker logs --tail=200 the-box on failure

Actions run logs on a public repository are readable by anyone. On a failed health gate, the workflow dumps the last 200 lines of the production app's logs. Pino request logs and boot errors can carry emails, session identifiers, internal URLs, or connection strings from stack traces — straight onto a public page. The same applies, more mildly, to docker compose ps output (internal port topology).

  • Write the log dump to a workflow artifact (artifacts inherit repo-collaborator access semantics less cleanly — on public repos they're downloadable by anyone too, so prefer the next option), or
  • keep logs on the box: docker logs … > /var/log/the-box-deploy-fail.log and print only a pointer + exit code in the public log, or
  • grep-filter to safe lines (migration/boot markers) before echoing.

F4 · Actions pinned to mutable tags Low

all workflows — @v3/@v4/@v6 tags

Every action is referenced by major tag. Tags are mutable: a compromised action repo can repoint v2 at malicious code. Highest-value targets here: softprops/action-gh-release@v2 (third-party, runs in the job holding contents: write) and docker/login-action@v3 / docker/build-push-action@v6 (see the DOCKERHUB_TOKEN secret — a hijack could push poisoned images under wifsimster/the-box:latest, which the homelab then auto-deploys). Pin all actions to full commit SHAs with the version as a trailing comment, and add Dependabot's github-actions ecosystem to keep them current.

F5 · Job outputs interpolated into shell Low

release.yml — ${{ needs.decide.outputs.new_version }} etc. inside run:

new_version derives from package.json's version string; on a patch bump, MAJOR/MINOR pass through verbatim. A crafted version like 1; curl …/.2.3 survives the split-and-increment and is then template-substituted into several run: scripts before bash parses them — textbook expression injection. Exploiting it requires write access to main (who could edit workflows anyway), hence Low — but it's free to fix and removes a class of surprise: pass outputs through env: blocks and reference them as "$NEW_VERSION" so bash sees them as data, never code. (inputs.version_type is a choice enum, so its inline use is safe.)

What's already good

  • ok Top-level permissions: contents: read in every workflow; only release.yml/publish elevates, and only to what it uses.
  • ok No pull_request_target, no checkout of PR head refs with secrets in scope.
  • ok No attacker-controlled event data (PR titles, branch names, issue bodies) in any ${{ }} shell interpolation; commit subjects in the changelog go to a file, never through eval-able shell.
  • ok deploy.yml correctly gates workflow_run on branches: [main] + conclusion == 'success'.
  • ok pages.yml HTML-escapes summaries and filenames before injecting into the index; Pages serves from a separate origin per the repo security policy.
  • ok Release concurrency serialized (cancel-in-progress: false), push failures fail loudly; buildx GHA cache scopes can't be poisoned from PR branches (cache isolation + ci.yml doesn't share scopes).
  • ok Bot-loop guard on decide is belt-and-suspenders on top of GITHUB_TOKEN's no-retrigger behavior.

Suggested order of remediation

  • 1. F1 repo setting ("all outside collaborators") — 1 minute, closes the worst path today.
  • 2. F3 log redaction — small deploy.yml edit.
  • 3. F2 --ignore-scripts + persist-credentials: false — small release.yml edit.
  • 4. F4 SHA pinning + Dependabot — mechanical, slightly noisy diff.
  • 5. F1 structural fix (ephemeral runner or pull-based deploy) — schedule as a task.
  • 6. F5 env-var hygiene — fold into the next release.yml touch.
