HTML-First Artifacts

Date 2026-05-10 · Attendees Tech-Writer · Architect · UX Designer · PM/Workflow · Facilitator Claude

We reviewed Thariq's "Unreasonable Effectiveness of HTML" and agreed: for visual or interactive artifacts (specs, plans, meeting records, design proposals, post-incident reports) prefer single-file HTML over Markdown. Markdown still wins for reference docs that tooling reads. This file itself is the first applied example.

1. Why this meeting

The Box already has a markdown meeting precedent at tasks/mobile-first-subagents-meeting.md. Thariq's argument is that as Claude's plans get longer, almost no one reads a 100+ line Markdown file end-to-end — but a self-contained HTML doc with SVG, tables, callouts and tabs gets read, gets shared, and lets the reader actually use the artifact (drag-drop ordering, sliders, copy-as-prompt buttons). We want that for our specs.

Crucial caveat from the article (and adopted here): do not turn this into a /html slash command or skill. It is a prompting practice, not tooling.

Naming/location

Mirror, don't fork: docs/<name>.html sits next to docs/<name>.md. Avoid a parallel docs/html/ tree — it splits discovery and rots.

PRD rule for tasks/

Markdown by default; HTML when the PRD has 2+ of {user flow, screen mock, state diagram, formula/curve}. The prd-rewards-* series is borderline-HTML; prd-e2e-test-coverage.md is firmly markdown.

Architect Repo integration & risk

What HTML unlocks here

  • Layered boundary visuals — colored bands with the "no inward import" rule physically drawn as a forbidden arrow, not just asserted in prose.
  • Interactive request trace — hover POST /api/guesses → game.service.submitGuess → guessRepository → guesses and the path lights up.
  • Queue topology with real cardinality — current Mermaid shows 1 generic worker; we have ~10 (import, sync, daily-challenge, geo-*, streak-risk-email, referral-announcement-email).
  • Socket.io room model — dynamic challenge_<id> rooms vs the singleton admin room, with event-name labels on the edges.

Pilot artifact spec

One killer SVG ~1200×600: three horizontal bands (Presentation/Domain/Infrastructure), animated dashed arrow tracing a real request flow, and a red strikethrough arrow for the forbidden inward dependency.

Lifecycle rule

Every HTML doc opens with a <!-- SUMMARY: … --> block (≤5 lines, plain text) — that's what a PR reviewer reads instead of the diff. Structural changes regenerate the file from a prompt rather than hand-edit.

Risk flagged. Inline <script> in committed docs is an XSS-shaped foot-gun the moment we serve docs/ from the Express app — Better Auth cookies are same-origin. Decide now that HTML docs are local-open / GitHub-rendered only, never mounted under our origin.

Reusable styles

Recommend: inline-only. The portability of single-file HTML beats the DRYness of a shared _artifact.css. If consistency drifts, regenerate from this meeting record's stylesheet.

Accessibility floor

  • Body text contrast ≥ 4.5:1; #ec4899 on dark fails for body — use only on ≥24px bold or as a non-text accent.
  • focus-visible outline preserved (2px #a855f7 ring); @media (prefers-reduced-motion: reduce) zeroes transitions.
Anti-pattern banned. Glassmorphism overload (backdrop-blur on every card, rainbow gradients, rounded-3xl). The Box is restrained dark gaming, not Vercel-marketing-page. One neon glow per page maximum, flat surfaces otherwise. No Bootstrap blue, no Tailwind default purple — use our #a855f7.
PM/Workflow Adoption rule, migration, success signal

The rule (paste-ready for .claude/CLAUDE.md)

Prefer single-file HTML (inline CSS, optional vanilla JS, SVG) for any artifact a
human will read and react to: PRDs in `tasks/`, subagent meeting notes, design
proposals, release notes, post-incident reports. Plain Markdown is still correct
for code-adjacent docs that tooling reads: `README.md`, `docs/*.md` reference,
ADRs, migration notes, anything rendered by GitHub by default. Do NOT create a
`/html` slash command or skill — prompt for HTML output explicitly per task.
Canonical examples: tasks/html-first-subagents-meeting.html, docs/architecture.html.

PRD migration policy

Migrate-on-meaningful-edit. The 9 existing PRDs are mostly historical/shipped — rewriting is busywork; we capture the upside the next time one genuinely changes.

One workflow win: PR review packets

For non-trivial PRs (tournaments, daily-login streaks, scoring tweaks) generate an HTML "review packet" alongside the PR description: state-machine SVG, before/after viewer markup, interactive scoring-tier slider, live-rendered Socket.io sequence. Doubles as the changelog entry.

Success signal (4 weeks)

At least 3 of the next 5 PRDs/specs landed in tasks/ are .html AND each has a non-author commenter who left feedback referencing an interactive element — not just prose. HTML files with no engagement = we're just reformatting Markdown.

Open question

Do HTML artifacts stay in-repo only (raw-file preview / local serve), or do we stand up a lightweight tasks/ GitHub Pages publish step so non-engineer / French-speaking stakeholders can open them by URL?

3. Decisions

1
Adopt HTML-first for visual/interactive artifacts. PRDs with flows/state/economy, meeting records, design proposals, review packets, post-incident reports. Markdown stays for reference docs (api.md, database.md, ui-tokens.md).
2
Pilot doc: docs/architecture.html. Mirrors docs/architecture.md, ships in the same commit as this meeting record. The killer SVG is the layered request-trace diagram.
3
Location convention: mirror, don't fork. docs/<name>.html next to docs/<name>.md. No parallel docs/html/ tree.
4
No /html skill. Per the article and unanimous vote — this is a prompting practice. Adding tooling would freeze a style choice that's still settling.
5
Single-file, inline everything. No external CSS/JS, no asset folder, no build step. Portability over DRYness. The first .html doc is the style guide.
6
Every HTML doc opens with <!-- SUMMARY: ... --> (≤5 lines). PR reviewers read that instead of the diff. Structural changes regenerate; do not hand-edit.
7
HTML docs are local-open / GitHub-rendered only. XSS Never serve docs/*.html from the Express app (same-origin Better Auth cookies). If we ever need URLs, do it via GitHub Pages with a separate origin.
8
Existing PRDs: migrate-on-meaningful-edit. Don't bulk-convert.

5. Resolved questions

  1. GitHub Pages publish? resolved · yes Added .github/workflows/pages.yml: pushes to main that touch docs/*.html or tasks/*.html deploy to wifsimster.github.io/the-box. One-time repo setup: Settings → Pages → Source = "GitHub Actions". The workflow generates an index page from each file's <!-- SUMMARY --> block.
  2. Audience? resolved · senior devs only The Pages site is noindex'd and badged "internal". HTML artifacts aren't drafted for non-engineer or French-speaking stakeholders — they get the existing markdown PRDs and the in-app UI.
  3. Still open: revisit migrate-on-meaningful-edit after a quarter, or commit to it indefinitely?

6. How we'll know it worked

In 4 weeks (week of 2026-06-07): at least 3 of the next 5 specs/PRDs in tasks/ are .html, AND each has at least one non-author comment referencing an interactive element (chart, slider, toggle, draggable list). If we get HTML files but no engagement, we're just reformatting Markdown — revisit.
