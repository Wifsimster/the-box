Personalized Evening Nudge

Date 2026-06-09 · Attendees PM/Growth · Backend/Architect · UX Writer/i18n · Privacy & Trust · Facilitator Claude

The evening reminder that pulls players back to today's daily challenge should be personalized with the current title holder — "Venez battre {X} qui a fait un score de {Y} aujourd'hui !" The data is one query away (leaderboardRepository.findByChallenge(id)[0]); the hard parts are channel choice, dedupe against the existing evening jobs, and not broadcasting an unmoderated user-chosen name to thousands of people.

The Box · 18:00

1. Why this meeting

The user asked: "la notification du soir pour inciter les joueurs à jouer sur la partie du jour devrait être personnalisée — comme inclure la personne qui détient le titre du jour." Before designing, we mapped what The Box does today:

  • Evening nudges are currently e-mails (Resend), scheduled as BullMQ recurring jobs in src/index.ts: streak-risk-email at 19:00 UTC ("Ta série de N jours est en danger") and relance-email at 17:00 UTC (unclaimed daily reward). Both gate on email_marketing_consent and carry a 20 h per-user cooldown column.
  • A full web-push stack already exists (pushService.sendToUser(userId, payload) → VAPID fan-out worker), opt-in per device — but no scheduled push uses the leaderboard leader.
  • The title holder is free: findByChallenge(challengeId)[0] is rank 1, deterministically tie-broken (score desc → fastest finish → user id), anonymous and catch-up sessions already excluded. Fields: displayName, totalScore, completedAt.

So this is a net-new personalization layered on the evening-reminder slot, not a tweak to one string.

2. The personas

Four agents ran in parallel against the real code. Their contributions are below; the synthesis is section 3.

PM / Growth Channel, audience, KPI

Channel — both, sequenced

Push is the primary vehicle: it reaches the large non-streak population, renders the personalized line natively on the lock screen, and is where "play before midnight" urgency lands. But opt-in is partial, so we'd strand engaged users → also inject the leader line into the existing 19:00 streak-risk email. No third e-mail.

Audience & dedupe

  • Push: opted-in devices, haven't played today (exclude catch-up/anon), account age ≥ 2 days.
  • Mutual exclusion: if a user qualifies for the streak-risk e-mail, suppress the leader push that night — loss-aversion converts harder than competition, and double-nudging burns goodwill. Relance (17:00, reward) stays separate.

Timing

Fire push at ~18:00 UTC (a touch later → the leader/score is more "real" and harder to beat) but no later than ~21:00 so there's a comfortable play window before the midnight reset. Email stays 19:00.

KPI & experiment

Primary metric: play-rate uplift (qualified-nudged users who complete today's daily). Secondary: CTR, D1 return. A/B: personalized-with-leader vs. generic "Le défi du jour t'attend", 50/50, 2 weeks. Guardrails: push opt-out rate, e-mail unsubscribe rate, and a "named-and-shamed" check — does being the leader correlate with that user opting out?

Fallback stance

Empty board → generic reminder, never fabricate a leader. Recipient is the leader → flip to title-defense or simply suppress (they already played).

Backend / Architect Job shape, edge cases, migration

One leader lookup per run

Add leaderboardService.getTodayLeader() → resolves today's challenge id, calls findByChallenge(id, 1), returns { userId, displayName, totalScore } | null. Call it once at the top of the job; never inside the recipient loop (N+1 on game_sessions).

// queue/workers/evening-nudge-logic.ts
const leader = await leaderboardService.getTodayLeader()   // ONCE
if (!leader) return { sent: 0, message: 'no leader yet' }  // empty board

const candidates = await findCandidates()  // not-played-today, consent, cooldown
for (const u of candidates) {
  if (u.id === leader.userId) continue     // don't tell them to beat themselves
  await sendOne(u, leader)
}

New worker, not an extension

Extending streak-risk-email couples two audiences and two cooldowns. Register a separate recurring evening-nudge job mirroring the streak block, with an explicit mutual-exclusion clause in the candidate query (verify against the relance worker's existing logic before merge).

Edge cases

  • Empty board → getTodayLeader() returns null → bail early.
  • Recipient is leader → .whereNot('id', leader.userId) in the query.
  • Ties at rank 1 → already deterministic; no work.
  • Perfect score → copy fn branches to a "personne ne l'a encore battu" framing.

Pure copy fn (domain, unit-testable)

// domain/services/evening-nudge-copy.ts — zero infra imports
buildEveningNudge(locale, { leaderName, leaderScore, isPerfect }): { title, body }

Idempotency & migration

Stamp last_evening_nudge_at = NOW() on enqueue success (push is fire-and-forget); the 20 h cooldown makes a BullMQ retry a no-op. Migration 20260610_add_evening_nudge_cooldown.ts adds nullable last_evening_nudge_at TIMESTAMPTZ to user, mirroring last_streak_risk_email_at.

Flagged. Three evening jobs now target overlapping audiences — the mutual-exclusion must live in the query, not be assumed. Keep the new Date() UTC-slice consistent with the other UTC crons; introduce no timezone here.

Tone / safety

  • Gender-neutral FR: infinitives only on the player — "coiffer au poteau", "passer devant", "te détrôner" — never the agreed past participle "détrôné·e".
  • Long names: truncate {{name}} to ~20 chars + ellipsis at the data layer (OS hard-truncates push titles at ~40 chars).
  • Plurals: render the unit via t('…pts', { count: score }) for the rare 1-pt case.
Privacy & Trust Broadcast risk, opt-out, moderation

Is this a new exposure?

The name+score pair is already public via GET /api/leaderboard/today, so it isn't a fresh disclosure of content. But context escalates it: the leaderboard is pull; a notification is push to thousands. Confirmed in the schema: there is no leaderboard hide / opt-out flag — a player cannot remove themselves from the board at all. That's the core gap.

Featuring opt-out — ship the column day one

Add one boolean feature_in_notifications (default true) on user. Do not overload email_marketing_consent — that governs receiving, not being mentioned. Column at launch (no backfill later); the profile toggle UI can be a fast-follow.

Offensive / impersonating names — BLOCKER

Putting a user-chosen string into outbound mail/push effectively endorses it under our brand, and there is no profanity/moderation filter today. Minimum bar: a server-side profanity/blocklist check on the leader's display_name at send time; on failure, fall back to generic copy. Do not launch personalized copy without this.

GDPR. Naming the leader to market to others processes the leader's personal data for our promotional purpose — public-leaderboard legitimate interest covers in-app display, not mass marketing. Safe stance: featuring requires opt-out availability and name moderation; document the legitimate-interest basis + easy objection.

Minimum viable safe version

Personalize only if name passes the profanity/blocklist check and feature_in_notifications ≠ false. Otherwise ship the score-only generic line ("Un joueur a fait {{score}} aujourd'hui — viens le battre !"), which is always safe.

3. Decisions

1
Both channels. Primary: a new evening push at ~18:00 UTC. Secondary: inject the leader line into the existing 19:00 streak-risk-email. No third e-mail.
2
New evening-nudge worker, not an extension of streak-risk. Mirrors the streak block; mutual-exclusion lives in the candidate query so a streak-risk user is not double-nudged.
3
One leader lookup per run. leaderboardService.getTodayLeader() returns { userId, displayName, totalScore } | null; called once, passed into the loop. Reuses the deterministic, anon/catch-up-excluding findByChallenge.
4
Pure copy fn in domain. buildEveningNudge(locale, {...}) in domain/services/, zero infra imports, unit-tested across leader / empty / self / perfect / flagged-name branches.
5
Edge cases. Empty board → generic, never fabricate. Recipient is leader → v1 suppress via whereNot('id', leader.userId) (defend-title copy is a fast-follow). Ties → already deterministic. Perfect score → "personne ne l'a encore battu" framing.
6
Privacy gate is a LAUNCH BLOCKER. blocker Personalize only if the leader's display_name passes a profanity/blocklist check AND feature_in_notifications ≠ false; otherwise score-only generic copy.
7
Ship the opt-out column day one. feature_in_notifications BOOLEAN DEFAULT true on user — separate from email_marketing_consent. Profile toggle UI is a fast-follow.
8
Cooldown & idempotency. New last_evening_nudge_at TIMESTAMPTZ column, 20 h window, stamped on enqueue. Migration 20260610_add_evening_nudge_cooldown.ts.

6. How we'll know it worked

Two-week A/B (personalized-with-leader vs. generic): the personalized arm shows a measurable play-rate uplift on qualified-nudged users without a rise in push opt-out or e-mail unsubscribe — and no correlation between being featured as the leader and that user opting out. If personalization moves nothing, we keep the cheaper generic copy and drop the name.
