Public API for Streamers

Date 2026-05-14 · Attendees PM/Workflow · Architect · Backend & Security · Streamer Advocate · DevEx · Facilitator Claude

Streamers want to wire The Box into their chat: !box commands, on-screen overlays that move with their score, and webhook-driven "score posted in chat" moments. We agreed on a read-only, opt-in, key-authenticated public surface at /api/public/v1, an SSE event stream for overlays, and HMAC-signed webhooks for chat-bot triggers. No guess submission over the API — the door to remote-control cheating stays shut.

1. Why this meeting

Twitch/YouTube streamers have started asking for two things in DMs and the Discord: "how do I show my Box score in chat?" and "is there a way for chat to play along?". The Box already has a private REST surface under /api/ guarded by Better Auth session cookies — that's the wrong shape for a streamer to paste into Streamer.bot, Nightbot, or an OBS browser source. We need a separate, narrower, key-authenticated surface designed for headless consumption by third-party tooling.

Equally important: the existing private API uses cookie auth on the same origin as the SPA (see docs/authentication.md). We are not opening that up. The public API ships as a parallel router with bearer tokens, separate rate limits, and a documented scope contract.

2. The personas

Five agents ran in parallel. Their unedited contributions are below; the synthesis is in section 3.

PM/Workflow Use cases & scope

The four jobs streamers are actually hiring this for

  1. Chat command — viewer types !box, bot replies "Wifsim is ranked #42 today with 7,820 pts (streak: 11 days). Play: thebox.app/d/abc".
  2. Live overlay — OBS browser source shows score ticking up screenshot-by-screenshot, current tier (Easy/Hard), and a time remaining ring. Pure cosmetic — no spoilers.
  3. Webhook → chat — when the streamer finishes a daily, the bot auto-posts the final score, rank delta, and a shoutout to anyone who guessed correctly first.
  4. Leaderboard shoutouts — at stream start, bot fetches yesterday's top 5 and welcomes any of them present in chat.

Out of scope (explicitly)

  • Guess submission over API. Anti-cheat surface area; would also let any bot farm scores. no
  • Screenshot byte download for non-streamers. Hot-linking /uploads/screenshots/* is fine, but a "give me today's first screenshot URL" endpoint feeds cheating tooling and we get nothing back. no
  • "Play-in-chat" Twitch extension. Nice eventually, but it's a real Twitch ext review process — after the API exists, not now.

Audience

We're sized for the top ~50 streamers who'd care, not "every Twitch user". Pricing: free for now, gate behind streamer-opt-in toggle. Revisit if a single account starts pulling >10k req/day.

Naming

Public-facing label: "The Box Streamer Kit". Internally: just "public API". The kit-vs-API split lets marketing talk about overlays/bots without dragging viewers into REST jargon.

Architect Surface, versioning, transport choices

URL namespace

Everything sits under /api/public/v1/ — mounted by a new packages/backend/src/presentation/routes/public.routes.ts module that does not share middleware with the private routers. No Better Auth session middleware on this tree; a thin publicApiKey() middleware replaces it.

SSE, not Socket.io, for overlays

Our internal real-time lives on Socket.io (docs/realtime.md), but for OBS browser sources SSE wins: one-way, no sticky session, no socket-id juggling, works with stock EventSource, survives Cloudflare buffering with a 15s heartbeat. The backend emits the same domain events into a small fan-out adapter that both the Socket.io leaderboard room and the per-streamer SSE channel subscribe to.

Versioning

v1 in the path, additive-only changes within v1, breaking changes require v2 alongside. Streamer integrations have a long tail; pinning is non-negotiable.

Response shape

Keep the existing private-API envelope: { success, data, error }. Inconsistency between public and private would force two clients in any future internal tooling that talks to both. ETag on all GETs; 304 support so Streamer.bot polling stays cheap.

Backend & Security Keys, scopes, rate limits, abuse

API key format

tb_pk_live_<28-byte url-safe random>     # production
tb_pk_test_<28-byte url-safe random>     # test mode (no live webhooks)

Prefix is searchable; tb_pk_ is the secret-detection regex partners can grep their repos with. We store only SHA-256 of the key — the user sees the plaintext once at creation and then never again. Last-used timestamp + last-used-IP are stored unhashed.

Header

Authorization: Bearer tb_pk_live_…
# OR for SSE (EventSource can't set headers):
GET /api/public/v1/streamers/wifsim/live?key=tb_pk_live_…

Query-param keys for SSE are an exception. Mitigate the resulting URL-logging risk by: (a) accepting keys only on the SSE endpoint as a query param, (b) logging requests to that path with the key redacted via request-logging middleware, (c) marking keys "rotate after exposure" in the dashboard with a one-click regenerate.

Scopes

ScopeGrants
read:publicAnonymous-equivalent reads: challenge, leaderboard, public profile.
read:selfOwn session state (/streamers/me, /streamers/me/today).
stream:selfSubscribe to own SSE channel.
webhooks:selfManage own webhooks.

Default new key gets all four for the owner's own slug. Cross-account access is impossible — a key owned by user A calling /streamers/B hits the same anonymous read path as keyless callers (no extra data leak). This keeps the key story simple: keys identify you for your own data, not a permission to read other people's.

Rate limits

TierPer-minutePer-daySSE concurrent
Anonymous (by IP)605,0000
Keyed (by key)60050,0003
Webhook delivery (per endpoint)——3 retries w/ exponential backoff, 24h dead-letter

Implementation: express-rate-limit backed by Redis (already in the stack for BullMQ). Headers per the IETF draft: RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset.

Webhook signing

X-TheBox-Signature: t=1715712000,v1=<hmac_sha256(secret, "{t}.{body}")>
X-TheBox-Event: session.completed
X-TheBox-Delivery: 01HX...

Stripe-style. We ship a verifyWebhook() snippet in the docs for Node, Python, and PHP — those three cover ~95% of the chat-bot ecosystem.

SSRF risk. Webhook URLs are user-supplied. Block on register: any private/loopback IP (10., 172.16-31., 192.168., 127., ::1, link-local, AWS metadata 169.254.169.254), DNS-rebinding-resistant — re-resolve at delivery, not at register. Reject non-HTTPS. Reject our own host. Cap response read at 4KB and 10s.

CORS

Access-Control-Allow-Origin: * for all GET on /api/public/v1 with Vary: Origin; no credentials, no cookies. The private API stays origin-locked to CORS_ORIGIN.

Streamer Advocate What chat-bot wiring actually looks like

Ground truth: how streamers will use this

Most won't write code. They'll paste a URL into one of three tools. The shape of our API has to survive those tools' UIs.

ToolHow it consumes usOur concession
Nightbot$(urlfetch json https://…) returns ~400 char chat lines.Provide a ?format=chat query that returns a single-line, pre-formatted string under 400 chars. Don't make Nightbot users write JSONPath.
Streamer.botFull HTTP + JSON support, custom triggers on webhook events, sub-actions for OBS scene switches.Ship a downloadable .sb recipe in the docs that wires session.completed → "post in chat + show overlay scene for 8s".
OBS browser sourceHTML page, vanilla JS, EventSource.SSE endpoint (above) + a hosted reference overlay at https://thebox.app/overlay/:slug?key=… they can drop in as a browser source URL. Don't make them host their own.

The killer "post in chat" payload

// GET /api/public/v1/streamers/wifsim?format=chat
"🎮 @wifsim · Today: 7,820 pts (#42) · Streak: 11d · Play: https://thebox.app/d/abc"

One line, emoji opt-in via ?emoji=0, short URL via ?short=1. This is the endpoint 80% of streamers will ever hit.

Discoverability

New page /streamers in the SPA (English + French): big "Generate API key" button (gated to opted-in users), copy-paste snippets for each of the three tools, and a live tester that hits the user's own profile and shows the response. Keys ship with a test variant that has the webhook system disabled so a brand-new user can't blast their friends' Discord while clicking around.

Opt-in is mandatory

Profile visibility on the public API is OFF by default. Settings → "Public Streamer Profile" toggle, with a clear privacy paragraph: "Enabling makes your slug, today's score, and your rank queryable by anyone holding any API key." No key, no toggle, no data. Privacy isn't an afterthought — most users of The Box aren't streamers and shouldn't be discoverable.

DevEx Docs, OpenAPI, sample integrations

Docs layout

  • docs/public-api.md — reference, grep-friendly, mirrors docs/api.md's tone. Markdown, not HTML — it's a tooling-read reference.
  • docs/streamer-kit.html — visual landing for streamers, embedded snippets + the Streamer.bot recipe download. Lives next to docs/architecture.html as another HTML-first artifact.
  • docs/public-api.openapi.yaml — single source of truth for the surface. CI step asserts the routes match the spec.

SDK question

No SDK at launch. Two reasons: (a) the surface is small enough that fetch + four endpoints isn't a barrier, (b) writing one official SDK obligates us to keep three (Node, Python, PHP) in sync forever. Ship the OpenAPI spec; let early adopters generate their own. Revisit if we see signal.

Quickstart that goes in the docs (Node, <20 lines)

import { EventSource } from 'eventsource'

const KEY = process.env.THEBOX_API_KEY
const SLUG = 'wifsim'

// One-shot chat command
const profile = await fetch(
  `https://thebox.app/api/public/v1/streamers/${SLUG}?format=chat`,
  { headers: { Authorization: `Bearer ${KEY}` } }
).then(r => r.json())
console.log(profile.data) // "🎮 @wifsim · Today: 7820 pts (#42) …"

// Live overlay
const es = new EventSource(
  `https://thebox.app/api/public/v1/streamers/${SLUG}/live?key=${KEY}`
)
es.addEventListener('screenshot.scored', (e) => {
  const { position, score, total } = JSON.parse(e.data)
  console.log(`${position}/${total} → ${score}`)
})

Sandbox

tb_pk_test_* keys talk to a permanent demo streamer slug boxbot whose session is replayed on a 10-minute loop. Lets integrators build and demo without waiting for the next real session to start.

3. Decisions

1
Read-only at launch. No guess submission, no screenshot bytes, no session start. The API observes, it doesn't drive. Revisit only with an anti-cheat plan.
2
Namespace: /api/public/v1/, own router (public.routes.ts), zero shared middleware with the private surface, additive-only within v1.
3
Bearer API keys in the Authorization header (tb_pk_live_… / tb_pk_test_…); stored as SHA-256; SSE accepts ?key= query param with redacted logging.
4
Opt-in profile visibility. "Public Streamer Profile" toggle in settings, off by default. No data is queryable until the user flips it.
5
SSE for overlays (not a second Socket.io namespace). Internal events fan out through a small adapter so the same source feeds Socket.io leaderboards and per-streamer SSE.
6
HMAC-signed webhooks (Stripe-style X-TheBox-Signature), HTTPS-only URLs, SSRF guard re-resolving DNS at delivery, 3-retry exponential backoff, 24h dead-letter queue.
7
Rate limits backed by Redis: 60/min anon, 600/min keyed; 3 concurrent SSE streams per key. IETF RateLimit-* headers on every response.
8
Cross-account reads stay anonymous-equivalent. A key never elevates access to someone else's data — only to your own. Simplifies the scope story to "keys identify you".
9
OpenAPI is the spec of record at docs/public-api.openapi.yaml. CI asserts routes match. No official SDK at launch — ship a 20-line quickstart instead.
10
Test sandbox: tb_pk_test_* keys talk to a demo slug boxbot on a 10-minute replay loop. Webhooks disabled in test mode so beginners can't blast real chatrooms.

7. Risks & mitigations

RiskMitigation
Cheat tooling reading screenshot URLs / live position.No screenshot endpoint; /today returns screenshotsDone count, not current screenshot id; SSE events redact target game name until session completion.
Webhook SSRF (user-supplied URLs hitting internal services).Block private CIDRs, AWS metadata IP, link-local; resolve DNS at delivery time; reject non-HTTPS and our own host; cap response read.
API-key leakage via SSE query param in logs / browser history.Redact in request-logging middleware; one-click rotate in dashboard; document the risk on the key creation page.
Cookie auth spillover (Better Auth cookies hitting the public router).Public router intentionally has no Better Auth middleware; integration test asserts /api/public/v1/* ignores session cookies entirely.
A single streamer's overlay hammering us at 10Hz.SSE push model (no polling required); per-key concurrent-connection cap of 3; 600/min ceiling.
Privacy: non-streamer users discoverable via the API.Opt-in default-off; profile toggle gates all public-API visibility for that user; slug is user-chosen, never an internal id.

8. Open questions

  1. French docs for the streamer kit? Most target streamers are English-speaking, but a few key FR streamers exist. Lean: ship EN-only at M3, translate at M3.5 if Discord asks.
  2. Twitch OAuth as a path to API keys? Tempting (auto-link Twitch handle → slug), but adds an OAuth provider to maintain. Defer to post-M3.
  3. Catch-up sessions in /today? They don't count for the leaderboard. Lean: include with a "countsForLeaderboard": false flag, let bot authors decide whether to surface.

9. How we'll know it worked

8 weeks after M3 (~2026-08-27): at least 5 streamers with active keys, at least 2 visible overlay or chat-command implementations (Twitch clip or VOD timestamp counts), and zero abuse incidents or privacy reports. If we get keys generated but no integrations live, the docs/recipes are the gap — not the API. If we get abuse, the rate-limit + opt-in story needs revisiting.
