PRD — Per-Screenshot Countdown Timer

Daily game · 45s per screenshot · outcome of the UX / Architect / QA design meeting

One-liner. Each daily screenshot now has a 45-second limit. A circular countdown ring sits top-left; when it expires the screenshot is a permanent miss (« Temps écoulé ! ») and play moves on. The answer is revealed with the other unfound games at the end.

1. Decisions locked

QuestionDecision
Visual formCircular ring (~48–56px) + numeric core, top-left, z-40 — mirrors the score panel shell so the two top corners read as a pair. Keyboard-immune (top-anchored).
On timeoutPermanent miss (« échec définitif ») — position locked, non-revisitable, revealed as unfound at game end.
EnforcementClient-only (v1) — no server cutoff. The speed multiplier already floors at 1.0× past 20s, so a late guess earns nothing extra anyway; a hard server cliff would risk the guarded scoring path.
Source of 45Plumbed through the API (ScreenshotResponse.timeLimitSeconds), default 45. The tiers.time_limit_seconds column already existed; we expose & bump it rather than hardcode.

4. Architecture

Derived, not stored

useCountdownTimer computes remaining = limit·1000 − (now − roundStartedAt) on a 250ms interval. No duplicate countdown in the store → correct for free across round change, navigation and resume (all re-stamp roundStartedAt).

Timeout action

A single latched effect (keyed on roundStartedAt, gated to status === 'in_progress') calls timeOutCurrentPosition(): marks timed_out, advances to the next playable position, or ends the game if none remain.

Resume budget (anti-reset)

Each position carries a persisted timeSpentMs accumulator. setScreenshotData flushes the segment you're leaving into the outgoing position, so navigating away and back resumes the remaining budget instead of resetting to 45s. No document.hidden pause within a segment (matches the server's elapsed measure). The timeout latch keys on currentPosition, and useGameGuess sends the accumulated time so the server's Math.max also neutralises the speed-multiplier reset.

The timed_out status

New PositionStatus member. It slots into existing navigation logic without special-casing: findNextUnfinished / canNavigateTo don't return it (locked), hasSkippedPositions ignores it (not offered for review), and hasVisitedAllPositions counts it as visited. It survives refresh via the persisted positionStates, and the backend reveals it as unfound at endGame.

6. Test strategy

  • Determinism. Playwright page.clock installed before navigation; runFor advances virtual time so a 45s flow runs in milliseconds. The ring renders without an opacity-entrance so visibility never depends on clock advancement.
  • E2E scenarios. ring visible at the limit; ticks to warning/critical; auto-advance at 0 moves to the next position; a correct guess before 0 stops it; mobile viewport.
  • Unit. computeRemainingMs (clamped, no negatives, clock-skew safe), getTimerPhase thresholds, formatCountdown.
  • No regression. Legacy daily-game specs dwell <45s per position, so seed=45 leaves them green; we deliberately do not install a global fake clock in them (it would freeze framer-motion entrances and hide elements).
