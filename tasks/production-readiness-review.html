Production Readiness Review

The Box · v2.91.0 · branch claude/review-box-app-blockers-1einj

Verdict: do not ship yet. Eleven release-blocking issues found across four independent audits (security, data integrity, deploy/infra, realtime/observability). The most damaging are reachable without authentication or special tooling: any browser session can join other users' Socket.io rooms and read their reward events, and any client can subtract real seconds from their guess time to inflate scores. None of the fixes are large; most are 1–20 lines. Estimated 2–4 engineer-days to clear all blockers, plus one round of E2E regression.

Scope & method

Four parallel audits, each scoped to a non-overlapping risk surface, reading the repo at HEAD of claude/review-box-app-blockers-1einj. Findings are reproduced verbatim where they cite specific code; severities use the same scale across audits:

  • Blocker — exploitable today, or will break the production deploy.
  • High — ship-blocking if you have any SLO or compliance commitment; safe to defer one release at most.
  • Medium — not exploitable in isolation, but degrades operability or invites future regression.

Hardening recommendations (CSP, helmet tuning, log shipping, APM, etc.) were excluded by design — this review only lists things that are actually broken or unsafe today.

Blockers

B1 — Socket.io /admin namespace accepts any connection Blocker

packages/backend/src/infrastructure/socket/socket.ts:28-48

The /admin namespace has no io.use() middleware. Any unauthenticated client can io('/admin').emit('join_admin') and will be joined to admin-room, which receives live BullMQ completed / failed payloads — including job returnvalue and error stacks (see B-H1 below).

Fix 
adminNamespace.use(async (socket, next) => {
  const session = await auth.api.getSession({ headers: socket.handshake.headers as any })
  if (!session?.user || session.user.role !== 'admin') return next(new Error('Unauthorized'))
  next()
})

B2 — /geo and /notifications let any client join user:<anyId> Blocker

socket.ts:227-231 (geo) and 264-267 (notifications)

Both namespaces accept join_user with a client-supplied userId and call socket.join(`user:${userId}`) without checking the session. Server-emitted events (emitRewardGranted, emitUserPremiumGranted, emitGeoTierUp) are then visible to any attacker who guesses or enumerates a target's user id. This is a textbook IDOR with a live websocket channel.

Fix 
socket.on('join_user', async (userId: unknown) => {
  const session = await auth.api.getSession({ headers: socket.handshake.headers as any })
  if (!session?.user?.id || session.user.id !== userId) {
    socket.emit('error', 'unauthorized'); return
  }
  socket.join(`user:${userId}`)
})

B3 — BETTER_AUTH_SECRET defaults to a known dev string Blocker

packages/backend/src/config/env.ts:13 — also visible in .env.example:25

Today the env loader does BETTER_AUTH_SECRET: process.env['BETTER_AUTH_SECRET'] || 'dev-secret-change-in-production-min-32-chars'. The fallback is exactly 32 chars, so the length check at lines 82-83 passes. A production rollout that forgets the env var boots with a publicly-known secret; an attacker who notices can forge session cookies for any user.

Fix 
// env.ts
BETTER_AUTH_SECRET: process.env['BETTER_AUTH_SECRET'],
// validateEnv()
if (!env.BETTER_AUTH_SECRET) throw new Error('BETTER_AUTH_SECRET is required')
if (env.BETTER_AUTH_SECRET.length < 32) throw new Error('BETTER_AUTH_SECRET must be ≥ 32 chars')
if (env.BETTER_AUTH_SECRET.startsWith('dev-secret')) throw new Error('Refusing to start with dev secret')

B4 — Round timer is client-supplied Blocker

presentation/routes/game.routes.ts:226 → domain/services/game.service.ts:450

roundTimeTakenMs is taken from the request body and fed straight into calculateSpeedMultiplier, which awards 0.5× to 2.0× on score. Any user can send { "roundTimeTakenMs": 1 } and pin themselves at the top of the leaderboard. This is the easiest exploit in the codebase and would invalidate every leaderboard run on day one.

Fix

Stop reading roundTimeTakenMs from the body. Record round_started_at server-side when the round becomes active and compute elapsed at submit. Until that refactor lands, the one-line mitigation is to cap the client value against a server reference: const elapsed = Math.max(serverElapsed, clientElapsed).

B5 — submitGuess mutates four tables without a transaction Blocker

domain/services/game.service.ts:555-634

A successful guess inserts a guess row, updates tier_sessions, updates game_sessions, decrements inventory (for hint / second-chance applications), and may insert achievements. None of this is wrapped in db.transaction(). Concurrent guesses (one user, two tabs / one bot) can: double-apply a second-chance floor, decrement the same inventory item twice, or commit a partial guess that leaves the session in an unrecoverable state.

Fix

Wrap the route handler body in db.transaction(async (trx) => { ... }) and thread the trx through the repositories that already accept it. Add a unique index on guesses(tier_session_id, position) if you want to make duplicate-correct submissions a hard error rather than just inefficient.

B6 — Daily-login claim has a check-then-act race Blocker

domain/services/daily-login.service.ts:278-342

The (user_id, claim_date_utc) unique index added in 20260503_* protects the claim row, but the surrounding logic — streak increment, inventory upsert, freeze auto-consume at lines 200-240 — runs outside a transaction with the insert. Two concurrent POST /daily-login/claim calls can both pass the !status.canClaim guard and double-credit inventory before the second insert fails.

Fix

Move the entire claim path (status read, freeze consume, streak update, inventory upsert, claim insert) into a single transaction. Let the unique constraint be the source of truth: catch the duplicate-key error and return the existing claim instead of running side effects again.

B7 — Completed session can be re-started Blocker

game.service.ts:362-365

startChallenge reads is_completed once and then creates a new tier_sessions row. Two rapid calls race past the check and a player gets a second authoritative run at the same daily, overwriting their original score.

Fix

Replace the read-check with an atomic guarded update or a unique constraint on (game_session_id, tier) for tier sessions. Cheapest patch: select-for-update inside the transaction added in B5.

B8 — No SIGTERM / graceful shutdown Blocker

packages/backend/src/index.ts (around the start() call)

Docker sends SIGTERM on rolling deploy and waits ~10s before SIGKILL. The current entrypoint registers no handlers, so in-flight HTTP requests are cut mid-response, Socket.io clients see a hard disconnect, BullMQ jobs are interrupted without state, and the Knex / Redis pools never drain. Every deploy risks a partial guess commit or a partial daily-login claim.

Fix 
const shutdown = async (signal: string) => {
  logger.info({ signal }, 'shutting down')
  httpServer.close()
  io.close()
  await Promise.allSettled([
    importQueue.close(), geoQueue.close(), pushQueue.close(),
    importWorker.close(), geoWorker.close(), pushWorker.close(),
    db.destroy(),
  ])
  process.exit(0)
}
process.on('SIGTERM', () => shutdown('SIGTERM'))
process.on('SIGINT',  () => shutdown('SIGINT'))

B9 — importQueue never trims completed jobs Blocker

packages/backend/src/infrastructure/queue/queues.ts:6-17

Both geoQueue and pushQueue set removeOnComplete in defaultJobOptions; importQueue does not. RAWG import jobs accumulate forever in Redis. With the current weekly import cadence this is slow burn, but with the daily-challenge worker living on the same queue (see B10) it bloats much faster.

Fix 
// queues.ts
export const importQueue = new Queue('import', {
  connection,
  defaultJobOptions: {
    attempts: 3,
    backoff: { type: 'exponential', delay: 5000 },
    removeOnComplete: { count: 100, age: 7 * 24 * 3600 },
    removeOnFail:     { count: 100 },
  },
})

B10 — Daily-challenge worker derives the date at execution time Blocker

infrastructure/queue/workers/daily-challenge-logic.ts (cron at index.ts:372-384)

The job is scheduled 0 0 * * * UTC, but the worker calls getTodayDateUTC() at run time. A container that restarts at 23:59:55 UTC re-registers the recurring job after midnight, so the job either fires for tomorrow's date (skipping today) or fires twice around the boundary. Without idempotency on (daily_challenge.date) this can create duplicate challenges or silently skip a day — both very visible bugs for daily players.

Fix

Capture the target date at enqueue time: importQueue.add('create-daily-challenge', { date: todayUtcISO }, { jobId: `daily-${todayUtcISO}` }). The jobId makes the queue de-dupe on its own. Add a unique index on daily_challenge.date if the schema doesn't already have one.

B11 — Backend source maps shipped in production image Blocker

packages/backend/tsconfig.json:15 + Dockerfile (copies dist/ wholesale)

"sourceMap": true emits *.js.map next to compiled JS, and the production image ships them. Anyone who pulls the public Docker Hub image (or hits a hypothetical /api/index.js.map) gets the original TypeScript with variable names, comments, and SQL strings. Combined with B3 this materially lowers the cost of a targeted attack.

Fix

Either flip sourceMap off in the production tsconfig, or add RUN find /app/packages/backend/dist -name "*.map" -delete to the runtime stage of the Dockerfile.

High severity

H1 — Admin socket events broadcast raw job payloads High

socket.ts:63-152

BullMQ completed / failed handlers forward the job's returnvalue and failedReason straight to admin-room. If a job ever surfaces a DB error, a stack trace, a RAWG response, or a user record, every connected admin sees it. After B1 is fixed this is reduced to a privilege issue rather than a public leak, but it's still worth scrubbing.

H2 — Wrong-guess response leaks future hints High

game.service.ts:720-770

The submit response includes availableHints covering the remaining games' release year, publisher, developer, and genre — even on a wrong guess. A player (or a bot) can throw away one guess to harvest hints for every subsequent screenshot in the run.

Fix

Only emit hints for the position that was just resolved correctly, or wait until the tier session is fully completed. Stop returning data about games the user hasn't reached yet.

H3 — guesses.guessed_text is nullable High

migrations/20260113_merged_schema.ts:290-303 + game.service.ts:442

A guess with an empty guessText short-circuits fuzzy matching and falls back to the client-supplied gameId. Combined with a leaked id (H2 is one source), a client can submit a "blank correct" guess.

Fix

Make the column NOT NULL and reject empty strings in the route Zod schema. Server-side, treat gameId as a tie-breaker, never a sole correctness signal.

H4 — First-user admin bootstrap race High

infrastructure/auth/auth.ts:240-249

SELECT COUNT(*) FROM "user" followed by an insert is not serialisable. Two simultaneous signups on a fresh DB can both become admin. Low blast radius after launch, but it is the kind of thing that surfaces only the one time it really matters.

Fix

Wrap the count + insert in a serialisable transaction, or add a partial unique index CREATE UNIQUE INDEX one_admin ON "user"(role) WHERE role = 'admin' and let the DB enforce it.

H5 — Knex pool: no SSL for remote DBs, no idle timeout High

packages/backend/knexfile.ts:20-22

Production block sets only min/max. RDS, Neon, Supabase — any managed Postgres — will reject the connection without ssl. Without idleTimeoutMillis the pool won't free idle clients across deploys.

Fix 
production: {
  client: 'pg',
  connection: {
    connectionString: env.DATABASE_URL,
    ssl: env.DATABASE_URL.includes('localhost') ? false : { rejectUnauthorized: false },
  },
  pool: { min: 2, max: 10, idleTimeoutMillis: 30_000 },
}

H6 — No process-level unhandledRejection handler High

packages/backend/src/index.ts

Without a process-level handler, an unhandled rejection in a worker is logged and silently drops the worker. The HTTP server keeps accepting traffic, so the queue stops draining without alarms. Fail fast and let Docker's restart policy bring the container back.

Fix 
process.on('unhandledRejection', (err) => { logger.fatal({ err }, 'unhandledRejection'); process.exit(1) })
process.on('uncaughtException',  (err) => { logger.fatal({ err }, 'uncaughtException');  process.exit(1) })

H7 — express.json() with no explicit limit High

backend/src/index.ts:78

Default Express limit is 100kb, which is mostly fine, but the value is implicit and any future app.use(express.json({ limit: ... })) change is a memory-DoS vector if set carelessly. Pin it.

Fix

app.use(express.json({ limit: '256kb' }))

H8 — Request logger doesn't strip auth tokens from URLs High

presentation/middleware/request-logger.middleware.ts:8-14

Password-reset and email-verification links include single-use tokens as query parameters. The middleware logs req.url verbatim, which means anyone with log access can read the token within its TTL.

Fix

Log req.path instead of req.url, and never serialize req.query or req.body from auth-adjacent routes.

Medium severity

M1 — Backup script exposes DB password via PGPASSWORD Medium

compose.yml:116 + scripts/db-backup.sh:12

PGPASSWORD is set as a container env var, visible in docker inspect and in any leaked log line that dumps env. Switch to a ~/.pgpass file mounted read-only at 0600.

M2 — Backup script has no fail-fast on pg_dump failure Medium

scripts/db-backup.sh:12-15

A piped pg_dump | gzip hides the upstream exit code, then find ... -delete deletes old backups regardless of whether the new one is valid. One failed backup can wipe the rotation window.

Fix 
set -o pipefail
pg_dump ... | gzip > "$F.tmp" && mv "$F.tmp" "$F" || { logger 'backup failed'; exit 1; }
# only run retention after a successful backup

M3 — Hardcoded TZ=Europe/Paris in compose.yml vs UTC crons Medium

compose.yml:28

Pino timestamps and any naive new Date() uses get Paris time, while the BullMQ crons explicitly use UTC. Pick one (UTC everywhere is the convention used by the rest of the codebase) and remove the override.

M4 — Streak-freeze auto-consume not transactional with the streak update Medium

daily-login.service.ts:200-240

Two concurrent /status calls on the first day after a missed day can both pass the "would reset, missed exactly one" guard and both try to consume a freeze. Mostly a refund/UX bug, but it shows up the moment a user has two tabs open.

M5 — Recurring-job dedup race on boot Medium

backend/src/index.ts:356-369

The boot path does removeRepeatableByKey then add for each cron. Two containers booting in parallel (rolling deploy with overlap) can interleave and end up with either zero or two recurring jobs. Use stable jobIds and idempotent add calls everywhere.

M6 — No Socket.io Redis adapter Medium

socket.ts:19-25

Not a problem for the current single-container deploy — flagging it only because the compose.yml structure suggests horizontal scale is a near-term goal. The day a second replica appears, rooms and emits stop working without an adapter or sticky sessions.

M7 — Admin access denials logged at warn Medium

middleware/auth.middleware.ts:92-94

With the typical production LOG_LEVEL=info these are dropped. Promote to error so admin-probing attempts are always visible.

M8 — Migration 20260113_merged_schema.ts not partial-failure safe Medium

migrations/20260113_merged_schema.ts:33-60

Seventeen createTable calls in a single migration. A failure on table #10 leaves the schema half-built and Knex won't re-run a partially committed migration. Either wrap the whole thing in a transaction (Knex supports this when none of the statements are DDL that breaks transactional DDL semantics on Postgres) or use createTableIfNotExists.

M9 — i18n fallback is implicit Medium

packages/frontend/public/locales/{en,fr}/translation.json

Both files have ~1879 keys, but the i18n setup doesn't pin a fallbackLng. The result is that a missing FR key shows the English string mid-page on a primarily French app — a visible quality regression, not a bug. Pin fallbackLng: 'fr' and audit for missing keys before launch.

Pre-ship checklist

Minimum bar to unblock launch:

  • B1, B2 — Socket.io /admin, /geo, /notifications all gated by a session-validating io.use() middleware.
  • B3 — BETTER_AUTH_SECRET default removed; missing-secret throws on boot.
  • B4 — Server-authoritative round timer (or at minimum a server-side cap that floors abuse).
  • B5 — submitGuess wrapped in a Knex transaction.
  • B6 — Daily-login claim path transactional; duplicate-key fallback path proven by a unit test.
  • B7 — startChallenge uses an atomic guarded update or a unique constraint.
  • B8 — SIGTERM handler closes HTTP, Socket.io, queues, workers, and DB pool.
  • B9 — importQueue has removeOnComplete / removeOnFail bounds.
  • B10 — Daily-challenge job carries its target date in the payload and uses a stable jobId.
  • B11 — Source maps stripped from the production image (or sourceMap: false in prod tsconfig).
  • H1–H8 — addressed or explicitly deferred with a tracking issue.
  • One full npm run build && npm run lint && npm test && npm run test:e2e pass on the fix branch.
  • Manual smoke: register two users, play a daily, claim a daily-login, hit the admin panel, force a worker restart, confirm graceful shutdown logs.

Areas verified clean

Surfaces examined during the audits where no blocking or high-severity issue was found:

  • SQL injection — no knex.raw / db.raw concatenating user input was discovered.
  • Zod validation — the routes audited all use the validation middleware; no route was found reading req.body without a schema.
  • Catch-up leaderboard filter — leaderboard.repository.ts consistently filters is_catch_up = false on today, percentile, and monthly queries (lines 29, 61, 98).
  • User cascade — game_sessions.user_id has ON DELETE CASCADE; deleting a user does not leave orphan rows in leaderboard sources.
  • Stripe webhook idempotency — two-phase claim / processed marking is in place; recommend simplifying to a single unique-constrained row but no exploit found.
  • CSRF — Better Auth's session cookie is SameSite=Lax by default and all state-changing routes are JSON-body POSTs, which is the standard mitigation; no cross-site exploitable surface found.

Re-run this review after the blocker patches land — the security and game-correctness audits will need a second pass once the transactional refactor in B5 / B6 / B7 is in place.
