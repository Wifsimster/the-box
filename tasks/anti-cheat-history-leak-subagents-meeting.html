Peut-on lire l'historique des autres avant d'avoir joué ?

Question posée · L'utilisateur a demandé une revue : un joueur peut-il consulter l'historique des autres joueurs avant d'avoir terminé son propre challenge quotidien, et donc tricher en lisant leurs réponses ?

Branche · claude/review-game-cheating-prevention-BMjpB · Format · réunion sous-agents (3 personas)

Verdict commun : oui, trivialement. La garde anti-triche existe (getPublicGameSessionDetails) mais elle est du code mort à cause d'un Date === string. Et même réparée, un POST /api/game/start + POST /api/game/end immédiat valide la condition is_completed=true sans le moindre guess, débloquant à la fois la lecture des sessions d'autrui et les unfoundGames de sa propre session — qui contiennent les 10 noms de jeux du jour.

Participants

  • SECURITY-REVIEWER — analyse les vecteurs d'attaque, le typage côté DB, les side-channels.
  • BACKEND-DEV — relit la garde, les repos, et les autres endpoints qui renvoient des sessions.
  • PRODUCT-QA — déroule les recettes utilisateur depuis l'UI (Leaderboard, profil public, History).

Le bug pivot — Date === string

La garde dans packages/backend/src/domain/services/user.service.ts:245-270 :

const challenge = await challengeRepository.findById(gameSession.daily_challenge_id)
const today = new Date().toISOString().split('T')[0]          // string 'YYYY-MM-DD'
if (challenge && challenge.challenge_date === today) {        // Date === string  ⇒  toujours false
  const requesterSession = requesterId
    ? await sessionRepository.findGameSession(requesterId, gameSession.daily_challenge_id)
    : null
  if (!requesterSession?.is_completed) {
    throw new Error('TODAY_CHALLENGE_NOT_COMPLETED')
  }
}

La colonne challenge_date est déclarée table.date(...) dans packages/backend/migrations/20260113_merged_schema.ts:231. challengeRepository.findById (challenge.repository.ts:27-31) ne caste pas la colonne en ::text. Aucun pg.types.setTypeParser(1082, ...) n'est enregistré dans le repo. Avec le parseur par défaut de node-postgres, OID 1082 (DATE) est renvoyé en Date JS. La comparaison Date === '2026-05-20' est donc toujours fausse, et la branche anti-triche n'est jamais exécutée.

Indice que l'équipe est consciente du piège ailleurs : game.service.ts:370-372 protège déjà ce même cas avec typeof challenge.challenge_date === 'string' ? … : challengeDate.toISOString().split('T')[0]. Le garde-fou a juste été oublié dans user.service.ts.

Recettes de triche exécutables aujourd'hui

Recette 1 — clic Leaderboard 1 clic

  1. Bob (connecté, n'a pas joué aujourd'hui) ouvre /leaderboard. L'onglet « Quotidiens » est par défaut sur aujourd'hui (LeaderboardPage.tsx:73-84).
  2. Chaque ligne porte un icône Eye et un tooltip « clickToView » (LeaderboardPage.tsx:337, 356) — l'UI invite explicitement à cliquer.
  3. Le clic appelle fetch('/api/leaderboard/session/${entry.sessionId}') (LeaderboardPage.tsx:183).
  4. Côté serveur, leaderboard.routes.ts:20 appelle getPublicGameSessionDetails → le === casse → la garde tombe → renvoi 200 + GameSessionDetailsResponse complète.
  5. Le ResponsiveDialog (LeaderboardPage.tsx:582-686) affiche pour chaque position le correctGame.name, dans l'ordre des screenshots.

Résultat : Bob voit les 10 jeux du jour avant d'avoir cliqué sur « Jouer ».

Recette 2 — start + end instantané 2 requêtes

  1. POST /api/game/start/:todayChallengeId — crée la session.
  2. POST /api/game/end {sessionId} — gameService.endGame (game.service.ts:975-1021) flippe is_completed=true sans exiger un seul guess. screenshotsFound peut être 0.
  3. Bob appelle GET /api/user/history/:sessionId sur sa propre session. buildSessionDetailsResponse (user.service.ts:166-184, 200) renvoie unfoundGames — un tableau Game complet (nom, slug, cover) pour chaque position non devinée, soit les 10 jeux du jour.

Résultat : Bob a les 10 réponses depuis sa propre session, sans même toucher à celle d'un autre joueur. Bonus : la condition is_completed=true est désormais remplie, donc même si la recette 1 était réparée, Bob est maintenant autorisé à lire les sessions d'autrui (et à scorer au leaderboard du jour avec un score forfaitaire de 0 — visible ou pas selon la logique d'inclusion).

Recette 3 — profil public → fetch console anonyme OK

  1. GET /api/user/public/:username (user.routes.ts:18) est sans authMiddleware et renvoie recentSessions: [{ sessionId, challengeDate, totalScore, completedAt }, …] (5 dernières sessions complétées, ligne :36-46, :82-87) — y compris celle du jour.
  2. L'UI affiche seulement date+score (PublicProfilePage.tsx:136-156), mais le sessionId est dans la réponse JSON visible dans DevTools → Network.
  3. Bob copie le sessionId du jour d'Alice et lance dans la console : 
    fetch('/api/leaderboard/session/<id>').then(r => r.json()).then(console.log)
  4. Même chemin que la recette 1 → réponses complètes.

Résultat : fonctionne même sans être connecté, donc impossible d'attribuer la lecture à un utilisateur.

Rapports des personas

SECURITY-REVIEWER — vecteurs & side-channels

1 — Bordure date / typage Exploitable

Voir le bug pivot. La garde est silencieusement neutralisée par le mismatch Date === string. game.service.ts:370-372 contient déjà la défense correcte, ce qui suggère que l'équipe a rencontré le piège ailleurs et l'a oublié ici.

2 — Course / forfait is_completed Exploitable

POST /api/game/end flippe is_completed=true sans contrôle de progression. game.service.ts:1017-1021 et :1099. Le drapeau confond « terminé » et « abandonné ». Encore plus court : la propre session du joueur expose les unfoundGames (objets Game complets) pour chaque position non devinée — c'est-à-dire toutes les positions s'il n'a pas joué.

3 — Catch-up Sain by design

isCatchUp est dérivé côté serveur (game.service.ts:374) à partir de la date du challenge, pas d'un champ envoyé par le client. La garde gate sur challenge_date === today, indépendamment de is_catch_up.

4 — Autres endpoints fuyants Mixte

  • GET /api/user/history/:sessionId — propriétaire-only mais renvoie unfoundGames (cf. recette 2).
  • GET /api/game/today & /api/public/v1/challenge/today — date + totaux seulement. Sain.
  • GET /api/game/image/:screenshotId — gated par userCanAccessScreenshot (screenshot.repository.ts:45) mais débloqué dès qu'il existe une session pour le jour, même vide.
  • Achievements & sockets : aucun nom de jeu dans les payloads vérifiés.

5 — Sidechannel sur le 403 Sain

leaderboard.routes.ts:27-43 renvoie un code d'erreur générique. Pas de fuite de challengeId/gameId, pas de différence de timing exploitable.

6 — Percentile Sain

GET /api/leaderboard/today/percentile?score=X renvoie {percentile, totalPlayers, rank}. Profile la distribution mais ne révèle pas le jeu.

7 — Forge de userId Sain

optionalAuthMiddleware (auth.middleware.ts:50-71) ne lit que la session Better-Auth résolue, pas un header/body. Pas de forge possible.

BACKEND-DEV — hygiène d'implémentation

Walkthrough getPublicGameSessionDetails

  • findCompletedGameSessionById (session.repository.ts:75-83) filtre bien sur is_completed = true. Pas de fuite de session en cours.
  • challengeRepository.findById (challenge.repository.ts:27-31) renvoie challenge_date comme Date JS (cf. bug pivot). findByDate caste en ::text mais pas findById.
  • findGameSession(userId, challengeId) (session.repository.ts:55-63) est correctement scopé sur les deux clés. Pas de match cross-utilisateur.
  • is_completed est flippé par endGame sur la route /api/game/end sans condition de progression — d'où la recette 2.

Endpoints alternatifs renvoyant une session

  • game.routes.ts : aucune route /results publique ne renvoie de game.name d'un autre joueur.
  • achievement.routes.ts : pas de leak observé.
  • admin.routes.ts protégé par adminMiddleware (auth.middleware.ts:73-104). Hors scope.
  • Sockets : émissions ciblées sur admin-room et user:${userId}. Pas de broadcast public avec nom de jeu.

Cache / CDN

Aucun Cache-Control: no-store sur /api/leaderboard/session/:sessionId ni sur /api/user/*. Si un proxy intermédiaire ajoute une heuristique, un 200 peut être servi de l'autre côté de minuit UTC. À durcir.

PRODUCT-QA — déroulés utilisateur depuis l'UI

Leaderboard page

LeaderboardPage.tsx : onglet « Quotidiens » par défaut sur aujourd'hui, lignes cliquables avec icône Eye et tooltip clickToView (:337, :356). Click → fetch('/api/leaderboard/session/${id}') (:183) → modale (:582-686) qui rend correctGame.name par position avec les ✓/✗ et les tentatives erronées via GuessAttemptsList. L'UI guide l'attaquant.

Profil public

PublicProfilePage.tsx:20-45, 136-156. Route backend sans authMiddleware. Les sessionId du jour sont dans la réponse JSON même si le DOM n'en montre rien.

History page

HistoryPage.tsx:91, 404-406 et GameHistoryDetailsPage.tsx:36 : propriétaire-only via findGameSessionById(sessionId, userId). Pas de surface UI pour voir l'historique d'un ami. Sain

Attaque en tab parallèle

Aucun verrou empêche d'ouvrir /leaderboard dans un second onglet pendant qu'on joue. Bob peut peeker entre chaque screenshot.

Bordure « hier vs aujourd'hui »

Comme la comparaison === est cassée tous les jours, la bordure de minuit n'est plus le problème pertinent : chaque jour est traité comme « passé ». À traiter en même temps que le bug pivot.

i18n

leaderboard.todayLocked existe en fr et en (translation.json:565). Le message s'affichera correctement quand la garde fonctionnera.

Actions recommandées

Avant tout commit de correctif : la branche actuelle ne contient aucune modification de code. Cette réunion est purement diagnostique. La décision quoi corriger et dans quel ordre est laissée à l'utilisateur.
  1. Réparer la garde (bug pivot). Soit caster challenge_date::text dans challengeRepository.findById, soit normaliser sur place avec le même guard que game.service.ts:370-372. La normalisation au repo est préférable : tous les consommateurs en profitent.
  2. Verrouiller endGame contre l'abandon-éclair. Refuser le flip is_completed=true tant qu'au moins un tier a été complété (pas seulement démarré), ou tant que screenshotsShown == totalScreenshots. Distinguer completed et forfeited dans le schéma serait plus propre — l'anti-triche utiliserait completed, le leaderboard utiliserait completed AND NOT forfeited.
  3. Strip unfoundGames tant que la session n'est pas vraiment finie. buildSessionDetailsResponse renvoie les noms des jeux non devinés — c'est pertinent pour un recap post-partie, pas pour un session-detail générique. Mettre la liste derrière un drapeau « tous les rounds joués » ou ne l'inclure que pour le propriétaire.
  4. Retirer sessionId de /api/user/public/:username, ou faire passer cet endpoint par optionalAuthMiddleware + même garde anti-triche. Le profil public n'a aucun besoin légitime d'exposer les sessionId.
  5. Ajouter Cache-Control: no-store sur /api/leaderboard/session/:sessionId, /api/user/history*, /api/user/public/*. Évite la substitution d'un 200 obsolète par un proxy.
  6. Test E2E anti-triche. Spec Playwright qui crée 2 users, fait jouer l'un, vérifie que l'autre reçoit 403 + TODAY_CHALLENGE_NOT_COMPLETED sur tous les chemins listés (leaderboard click, fetch direct, profil public → fetch). Doit échouer aujourd'hui, passer après correction.
