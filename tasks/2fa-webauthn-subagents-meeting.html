Double authentification & biométrie WebAuthn

Date 2026-05-20 · Branche claude/add-2fa-biometric-C7g8W · Présents Architect · Security · UX Designer · PM/Workflow · Facilitateur Claude

On ajoute un second facteur à The Box. Better Auth 1.4.10 expose déjà tout ce qu'il faut : le plugin twoFactor pour le TOTP + backup codes, et le plugin passkey pour la WebAuthn API (l'« API d'authentification HTML par empreinte digitale » du brief — c'est navigator.credentials côté DOM, qui adresse Touch ID, Face ID, Windows Hello, Android biométrique et les clés FIDO2). On active les deux plugins, on rend la 2FA obligatoire pour les admins, opt-in pour les joueurs.

1. Pourquoi cette réunion

Aujourd'hui l'auth de The Box est strictement e-mail + mot de passe (voir docs/authentication.md et packages/backend/src/infrastructure/auth/auth.ts). Le premier utilisateur enregistré devient automatiquement admin (cf. databaseHooks.user.create.before) — un mot de passe compromis sur ce compte donne accès au panneau d'administration, à l'import RAWG, à la gestion des utilisateurs et à la file BullMQ. Le risque n'est pas théorique : on a déjà un admin_audit_log (migration 20260506_admin_audit_log.ts) qui tracerait l'incident après coup, pas avant.

Le brief demande deux choses, qui sont en fait deux facteurs séparés :

  • 2FA TOTP — Google Authenticator, 1Password, Authy. « Ce que je sais » + « ce que je possède ». Couvre 95 % des comptes existants sans matériel spécifique.
  • WebAuthn / passkey — empreinte digitale, Face ID, Windows Hello, YubiKey. « Ce que je suis » + « ce que je possède ». Résistant au phishing par construction (challenge signé lié à l'origine), pas seulement un OTP relayable.

L'option par défaut « ne faire que TOTP » est rejetée d'entrée : nos joueurs sont majoritairement sur mobile (cf. useIsMobile, layout mobile-first), or le geste « ouvrir Authenticator → recopier 6 chiffres en 30 s » est exactement le frottement qu'on cherche à éviter. Touch ID / Face ID résout ça en un tap.

Architect Intégration Better Auth, migrations, surface API

Ce que Better Auth 1.4.10 nous donne gratuitement

  • Plugin twoFactor — TOTP (RFC 6238), backup codes (10 codes hex, usage unique), endpoint POST /api/auth/two-factor/enable, verify-totp, verify-backup-code. Crée les tables twoFactor et étend user avec un booléen twoFactorEnabled.
  • Plugin passkey — wrapper WebAuthn complet. Endpoints d'inscription (generate-registration-options → verify-registration) et de connexion (generate-authentication-options → verify-authentication). Crée la table passkey (1 ligne par credential).
  • Côté client authClient.twoFactor.* et authClient.passkey.* sont déjà dans better-auth/client/plugins. Pas de signature WebAuthn à manipuler à la main : la lib appelle navigator.credentials.create() / get() pour nous.

Diff serveur minimal

// packages/backend/src/infrastructure/auth/auth.ts
import { twoFactor, passkey } from "better-auth/plugins";

plugins: [
  username({ /* ... */ }),
  anonymous({ /* ... */ }),
  admin({ /* ... */ }),
  twoFactor({
    issuer: "The Box",            // libellé dans Authenticator
    skipVerificationOnEnable: false,
    backupCodes: { amount: 10, length: 10 },
  }),
  passkey({
    rpID: new URL(env.API_URL).hostname,  // ex. "thebox.example.com"
    rpName: "The Box",
    origin: env.CORS_ORIGIN,              // doit matcher exactement
  }),
],

Schéma — qui crée quoi ?

Better Auth génère ses propres migrations via npx @better-auth/cli generate — exactement comme pour les tables initiales user / session / account / verification. On commit le fichier généré sous packages/backend/migrations/20260520_twofactor_passkey.ts pour rester cohérent avec notre convention date-préfixée (le docker-entrypoint.sh les enchaîne au boot, donc pas de magie à ajouter).

Impact Clean Architecture

Aucun service du domain/ à toucher : la logique 2FA/WebAuthn vit entièrement dans la couche infrastructure (plugin Better Auth) et la couche présentation (routes auto-exposées sous /api/auth/two-factor/* et /api/auth/passkey/*). Le middleware authMiddleware reste identique — session.user.id ne change pas, c'est le chemin pour obtenir la session qui s'allonge d'un step quand 2FA est activé.

Ce qu'on ne fait pas

  • Pas d'OTP par SMS. Trop coûteux, trop attaquable (SIM swap), pas dans Better Auth de toute façon.
  • Pas de plugin email-otp en facteur principal — uniquement comme filet de récupération (cf. Security).
  • Pas de gestion manuelle de la table passkey. Les tentations de « lister les credentials dans un repository custom » conduiront à des bugs de cohérence avec l'auto-incrément interne du plugin.

Matrice de récupération

Le pire scénario fonctionnel d'une 2FA : l'utilisateur ne peut plus se connecter parce qu'il a perdu son téléphone et les codes de secours. On documente trois chemins, par ordre de préférence :

  1. Backup codes (TOTP) — 10 codes hex montrés une seule fois à l'activation, à imprimer / coller dans un password manager. Endpoint POST /api/auth/two-factor/verify-backup-code.
  2. Passkey supplémentaire — l'utilisateur peut enregistrer plusieurs passkeys (laptop + téléphone). On recommande explicitement d'en avoir au moins deux dans l'UI.
  3. OTP e-mail de secours — uniquement déclenchable sur la page de connexion 2FA après 24 h d'attente forcée (rate-limit côté Better Auth + log dans email_log). Empêche un attaquant qui a le mot de passe de bypasser instantanément la 2FA via la boîte mail compromise dans la même attaque.

Le cas admin-forcé

Migration 20260521 garantit un seul admin (one_admin_role_idx). Si cet admin se verrouille complètement, il n'y a personne pour déverrouiller. Donc :

  • L'activation admin impose de générer les backup codes et de confirmer leur téléchargement avant que le flag twoFactorEnabled ne soit committé.
  • Un script d'urgence en packages/backend/scripts/admin-reset-2fa.ts (exécution manuelle dans le conteneur, journalisé dans admin_audit_log avec une raison obligatoire) qui efface twoFactorEnabled et révoque les passkeys d'un user donné. Pas exposé via HTTP.
  • L'usage de ce script est tracé : admin_audit_log.action = 'EMERGENCY_2FA_RESET', valeur before/after JSON, signé par l'opérateur Linux qui l'exécute ($USER du conteneur).

Anonymous / guest

Les comptes anonymes (isAnonymous: true, plugin anonymous) n'ont pas accès à /settings/security. La 2FA suppose un canal de récupération (e-mail), ce que ces comptes n'ont pas. Le bouton du menu est masqué pour eux ; tentative d'appel direct → 403.

Risque opérationnel. Activer la 2FA forcée sur des admins existants sans flow d'enrôlement préalable les déconnecte définitivement. Le rollout doit être :
  1. Déployer le code, plugins activés mais twoFactorEnabled = false par défaut.
  2. Bannière persistante côté admin : « activez la 2FA avant le YYYY-MM-DD ».
  3. Job BullMQ programmé qui vérifie sans bloquer pendant 14 jours.
  4. Après 14 jours, middleware admin force la redirection vers /settings/security/enroll pour tout admin sans 2FA. Pas d'auto-déconnexion brutale.

Rate-limiting

Better Auth a son rate-limiter intégré (mémoire ou Redis). Avec Redis déjà présent (BullMQ), on l'active sur les routes two-factor/verify-* : 5 tentatives / 15 min / IP+user. Au-delà → verrouillage 30 min + log dans admin_audit_log (type 2FA_LOCKED) pour qu'on puisse détecter un brute-force coordonné.

UX Designer Flows enrôlement & connexion, sans friction superflue

Principe directeur

Une 2FA mal designée fait fuir les joueurs occasionnels. Trois leviers anti-friction :

  1. Passkey-first. Sur LoginPage, en plus du formulaire e-mail/mdp, un bouton primaire « Se connecter avec une passkey ». Un seul tap déclenche Touch ID. Personne ne tape un mot de passe s'il a déjà enregistré une passkey.
  2. Conditional UI. L'attribut autocomplete="username webauthn" sur le champ e-mail propose les passkeys du device dans la barre d'auto-fill du navigateur — pas de bouton à cliquer du tout sur Safari / Chrome récents. Better Auth supporte le discoverable login.
  3. Trust device. Coche optionnelle « Faire confiance à ce navigateur pendant 30 jours » sur la vérification TOTP. Implémenté via le flag natif twoFactor.trustDevice du plugin (cookie signé HMAC-SHA256, expire en 30 j, révoqué globalement quand l'utilisateur change son mot de passe).

i18n

Toutes les chaînes utilisateur sous public/locales/{fr,en}/security.json. Pas d'inline texte dans les composants. La page de paramètres reste français-default, anglais supporté, conformément à notre convention.

Accessibilité

  • Le champ TOTP : inputmode="numeric", autocomplete="one-time-code", longueur 6, auto-submit au dernier chiffre. Lecteur d'écran annonce « code à 6 chiffres ».
  • L'écran d'affichage des backup codes : bouton « copier dans le presse-papier » + bouton « télécharger en .txt ». Pas de QR pour les backup codes, ça ne sert à rien.
  • Animations de succès respectent prefers-reduced-motion (déjà imposé par la convention).

Les phases 0 → 3 sont incluses dans la même PR (feature gating par flag force2faForAdmins dans config/env.ts, OFF par défaut, flippé en prod après la phase 2). La phase 4 est une PR séparée.

Métriques

  • Sécurité. % d'admins avec 2FA. Cible J+30 : 100 %.
  • Adoption joueurs. Ratio comptes avec ≥ 1 passkey OU TOTP, parmi les comptes actifs sur 30 j. Cible J+90 : 15 %.
  • Friction. Taux d'échec de connexion à l'étape 2FA (verify-totp 4xx / total tentatives). Cible < 5 % en régime stable. Si > 10 %, c'est le wording ou le rate-limit qui pose problème.
  • Récupération. Nombre d'usages de backup codes / mois. Si nul, soit personne n'oublie son téléphone (improbable), soit ils ne savent pas que ça existe — déclencher une amélioration UX.
  • Support. Tickets « impossible de se connecter » — segmenter avant/après. Augmentation significative ⇒ pause sur la phase 3.

Communication

  1. In-app. Banner non-bloquante en haut du Header pour les comptes éligibles, lien direct vers /settings/security. Dismissable, réapparaît au bout de 7 jours.
  2. E-mail transactionnel via Resend (template renderEmailHtml existant), envoi unique au début de la phase 1. Sujet : « Protégez votre compte The Box en 30 secondes ».
  3. Confirmation d'activation — e-mail systématique « 2FA activée sur votre compte » avec date, IP géolocalisée, navigateur. Si l'utilisateur ne reconnaît pas → lien de révocation immédiate.

3. Décisions

1
On active twoFactor ET passkey, simultanément, dans le même bloc plugins: [...] de auth.ts. Pas de feature flag par plugin — les deux ou aucun.
2
Opt-in pour les joueurs, forcé pour les admins, gated par config.force2faForAdmins. Phase 3 du rollout retourne le flag.
3
Trois chemins de récupération : backup codes (par défaut, imposés à l'enrôlement), passkeys multiples (recommandées dans l'UI), OTP e-mail différé 24 h (filet de sécurité, rate-limité).
4
Migration générée par @better-auth/cli, committée comme packages/backend/migrations/20260520_twofactor_passkey.ts. Le docker-entrypoint.sh l'applique automatiquement au boot.
5
Passkey-first sur LoginPage : bouton primaire « Se connecter avec une passkey » + autocomplete="username webauthn" sur l'e-mail pour le Conditional UI natif.
6
Pas de SMS, pas de plugin email-otp en facteur principal. L'e-mail sert uniquement de chemin de récupération différé, jamais de facteur direct.
7
Anonymous / guest exclus. Endpoint two-factor/enable et passkey/register → 403 si isAnonymous.
8
Script d'urgence admin-reset-2fa.ts hors HTTP, exécution manuelle dans le conteneur, trace dans admin_audit_log avec raison obligatoire. break-glass
9
Rate-limit 5 tentatives / 15 min sur les routes verify-* via le rate-limiter Better Auth backed par Redis (déjà présent pour BullMQ).
10
E2E Playwright via page.context().addInitScript + Virtual Authenticator CDP pour passkeys ; TOTP testé avec un secret fixe seedé dans e2e-seed.ts + génération du code via otplib côté test.

5. Questions résolues & à trancher

  1. SMS comme second facteur ? résolu · non Coût, attaquable (SIM swap), pas dans Better Auth. Ferme.
  2. Forcer la 2FA aux joueurs avec un long streak ? phase 4 · séparée Pas dans le périmètre de cette branche. À discuter quand on aura la télémétrie d'adoption de la phase 1.
  3. Plugin email-otp en facteur principal ? résolu · non L'e-mail reste un canal de récupération, pas un facteur. Un attaquant qui a le mot de passe a typiquement compromis l'e-mail dans la même opération.
  4. Mode anonymous + 2FA ? résolu · incompatible Bloqué côté backend (403) et masqué côté frontend.
  5. Stocker l'IP de connexion par device pour notification de session ? ouvert Bonus de sécurité, mais pas dans cette PR. À mettre en backlog si la 2FA est largement adoptée.
  6. Que faire si l'utilisateur change son API_URL en prod (passkeys cassées) ? résolu · documenté Note explicite dans docs/better-auth-setup.md. rpID et origin sont immuables sous peine d'invalider toutes les passkeys.

6. Comment on saura que c'est réussi

À J+30 après merge : 100 % des comptes admin ont la 2FA activée, taux d'échec verify < 5 %, zéro incident de support « impossible de me reconnecter » non résolu en < 24 h.

À J+90 : ≥ 15 % des comptes actifs sur 30 j ont une passkey OU TOTP. Si on est à < 5 %, c'est que le flow d'enrôlement n'est pas découvrable — on déclenche une bannière non-bloquante en haut du Header sur les comptes éligibles, puis on remesure à J+120.
