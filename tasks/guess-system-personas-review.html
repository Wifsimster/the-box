Guess System — 12-Persona Review

Classic-mode scoring · fast-guess · bonus · partial-match · proximity hint · 12 subagent personas · 2026-06-14

One-line: the guess features are well-built in isolation, but three load-bearing things are broken or missing: the speed timer is forgeable (every leaderboard score is suspect), the configurable bonus multiplier is dead code (the "bonus" the owner asked about does nothing), and the freshly-merged proximity hint is a free answer-oracle because POST /guess has no rate limit and a wrong guess costs 0.

P0 — must fix

1. The speed timer is forgeable → trivial perfect runs

Speedrunner F1 · Competitive #1 · evidence: game.service.ts:564 (markRoundStarted on every non-prefetch getScreenshot), session.repository.ts:149-157, scoring game.service.ts:693-695

markRoundStarted re-stamps round_started_at = now() on every screenshot fetch, and the server only measures the current segment. effectiveTimeTakenMs = Math.max(serverElapsedMs, clientMs) therefore collapses to trust-the-client after any navigation. Take unlimited time to identify the game, re-fetch the screenshot, submit in <3 s server-elapsed → guaranteed 2.0× = 200 pts every time. The >2 s divergence check only logs. The "pause/resume time budget" is client-side only; the server has no accumulation.

Fix: accumulate active time per position server-side (bank serverElapsedMs on each fetch/submit), score off that, and never re-stamp the active position (UPDATE … WHERE round_position <> position OR round_started_at IS NULL). Reject (not just log) clientMs < serverMs beyond tolerance.

2. The advisory lock doesn't make submitGuess transactional

Backend P0 · evidence: session.repository.ts:139-144

withTierSessionLock takes pg_advisory_xact_lock on a trx, but the submitGuess body runs every repository call on the module-global db pool — not the locking transaction. Concurrency between two callers is serialised (both contend on the advisory key), but there is no atomicity: if saveGuess commits and updateTierSession then throws, the guess row survives with desynced counters. The comment ("releases on commit / rollback") implies a transaction that isn't there.

Fix: thread trx through the body (and read getCorrectAnswersCount on the same trx so the completion tally stays correct — see Backend P1), or drop the atomicity claim and document the recovery contract.

3. Timing out a screenshot reveals nothing — silent, confusing, permanent

Casual #1 · QA #7 · evidence: hooks/useRoundTimer.ts:122-130, stores/gameStore.ts:482-542

On timeout the player gets only a fleeting "Time's up!" toast, then is yanked to a new screenshot — no ResultCard, no correctGame, no "+0", and the position is a non-revisitable permanent miss that's never explained. For a first-timer a screenshot just vanishes. No onboarding mentions the 45 s clock or that running out is unrecoverable.

Fix: show the ResultCard in a "timed-out / +0" state revealing the answer (like a wrong guess) before advancing; add one onboarding line about the per-screenshot clock.

P1 — high

4. bonus_multiplier is dead code end-to-end (the "bonus" question)

Scoring #1 · Speedrunner F2 · Competitive #2 · QA #5 · evidence: game.service.ts:592 (read & shipped) vs game.service.ts:711-722 (scoring never references it); frontend grep = 0 hits

The column exists (migrations/20260113_merged_schema.ts:254, DECIMAL(3,2) default 1.0), is seeded, parsed, and shipped to the client in ScreenshotResponse.bonusMultiplier — but submitGuess scores only BASE_SCORE × speedMultiplier (cap 200, then partial factor). A "1.5×" screenshot scores identically to 1.0×. The frontend never reads it either. docs/database.md:277 calls it "Score multiplier"; docs/game-flow.md never multiplies it in. Compounding trap: the fastest exact guess already saturates the 200 cap, so even if wired before the cap a fast bonus is clamped straight back to 200 — there is no headroom for a bonus to ever matter on a fast play.

Fix (decide intent): either (a) wire it in (score = round(BASE × speed × bonus)) and redefine the cap as 200 × bonus so it has headroom; or (b) remove it from the response/types/DB and mark the column deprecated. Don't leave it half-plumbed. Add a scoring test with a non-1.0 multiplier either way.

5. Proximity hint is a free, unthrottled answer-oracle

Security F1/F2 · Backend P2 · evidence: game.routes.ts:302 (no rate limiter), game.service.ts:49 (WRONG_GUESS_PENALTY=0), :1048-1070 (hint on every miss), guess-proximity.service.ts:110-123

A wrong guess costs 0 points, has no attempt cap, isn't throttled, and returns a proximityHint. A bot turns each miss into a yes/no probe — "is the answer's developer X? publisher Y? same franchise?" — and binary-searches the studio, then enumerates that studio's titles (via the also-unauthenticated /games/search) to brute the answer, all before solving, at no cost. Crucially the relation flag itself (same_developer) is a one-bit confirmation about the answer, independent of the value the anti-leak contract reasons about.

Fix: rate-limit POST /guess per user (mirror the push routes); gate the hint behind the same "one honest attempt first" rule as the letter reveal and cap hints per position; treat the relation as sensitive (consider dropping same_developer/same_publisher, which leak most and teach least). Also throttle/auth /games/search.

6. Second-chance floor (70) overrides the partial discount (max 80)

Backend P1 · Scoring #3 · QA #3 · evidence: game.service.ts:719-787 (ordering: ×0.4 partial → letter penalty → floor 70)

A partial maxes at 80 and can fall to ~28. The second-chance floor lifts any sub-70 score to 70 after the partial reduction — so a deliberately discounted franchise-only guess ends at 70, near the 80 ceiling, and the floor silently negates the 0.4× the rest of the system works to preserve. The floor is precision-blind. Also untested: the harness wires both findPending mocks to null, so partial×floor and partial×letter-penalty never run.

Fix: scale the floor by PARTIAL_MATCH_FACTOR for partials (or apply the partial factor after the floor), or explicitly document that second-chance erases the partial penalty by design. Add the combination tests.

7. Proximity hint is persisted to localStorage → stale layout on revisit/refresh

Mobile #1 · evidence: useGameGuess.ts:92-95 (writes into positionStates), gameStore.ts:735 (in partialize), GuessInput.tsx renders from positionStates[pos]?.proximityHint

The "warmer" hint is stored per-position and only cleared on a correct guess, and it's persisted. Skip away and back — or reopen the PWA — and the banner reappears below the input, growing the dock and shoving the screenshot up, with no active guess to justify it. It's ephemeral feedback being treated as durable state.

Fix: clear proximityHint on navigation/skip, or strip it from partialize, or gate the render on "this is the active in-progress position."

8. Lone roman-letter titles ("Pokémon X, Y") get a bogus series number → exact false positives

Fuzzy #1 · evidence: fuzzy-match.service.ts:87 (roman regex includes bare X/V/I), parseGameTitle:216

extractSeriesNumber("Pokémon X, Y") matches the standalone X → seriesNumber = 10. The exact/partial guard then hinges on a phantom "10". "pokemon 10" could grade exact for a game that has no number 10. Same hazard for Final Fantasy X / X-2, Kingdom Hearts X. Because a partial/exact now awards points and ends the round (Fuzzy #2), a grading false-positive is a scoring bug, not a cosmetic one.

Fix: in extractSeriesNumber, don't treat a lone single-letter roman as a number when it's adjacent to a comma / another single letter, or only extract romans that follow a series-name token. Add the Pokémon X,Y and FF X cases to the matcher tests.

9. "Live leaderboard with real-time updates" isn't wired

Competitive #3 · evidence: docs/realtime.md:73,122 document leaderboard_update; grep across the app = 0 emitters; LeaderboardPage.tsx:163,202 fetch once, no socket .on / no polling

The board is a one-shot fetch despite the marketed "Socket.io real-time updates." During a contested day players see a stale snapshot until manual refresh.

Fix: emit leaderboard_update into a challenge_${id} room on session completion and subscribe on the page — or correct the docs/marketing claim.

10. Two a11y announce gaps in the core loop

a11y #1, #2, #3 · evidence: GuessInput.tsx:98-101, ResultCard.tsx:176-287, CountdownTimer.tsx:38-44

  • Wrong-guess announcement is suppressed for reduced-motion users — the aria-live text only renders when isShaking, which is gated behind !prefersReducedMotion. A screen-reader + reduced-motion user gets no miss feedback. Fix: drive the live region from the result, gate only the animation.
  • Result card isn't a dialog and isn't announced — no role, no focus move, outcome/score/answer never reach AT. Fix: role="dialog" aria-modal + focus the Next button + a role="status" outcome line.
  • Countdown is silent to AT (role="timer" ⇒ live=off): the permanent-miss clock is sighted-only. Fix: a polite region announcing coarse milestones (30 s / 10 s / time-up).

11. The two new features emit zero analytics — impact is unmeasurable

Analytics #1, #2, #3 · QA · evidence: game.service.ts:864-876 (guess_submitted payload omits matchPrecision & proximity), ProximityHintBanner.tsx / ResultScoreDisplay.tsx fire no GoatCounter event

isCorrect collapses exact and partial into one boolean, and the guesses table has no match_precision column, so partial-rate is lost at write time. Proximity firing/relation is never recorded server- or client-side. The features could be returning null 100% of the time in prod and nobody would know.

Fix: add matchPrecision + proximityFired/relation + attemptNumber to the guess_submitted payload (and a schemaVersion marker for the cutover); fire a minimal, consent-gated GoatCounter event on banner/badge impression.

12. Guess-error toasts are hardcoded English (default locale is French)

i18n #1 · evidence: lib/errors.ts:204-230 returns English literals → useGameGuess.ts:166 → GuessInput.tsx:86 toast.error; also 'Session not initialized' at useGameGuess.ts:39

A French player hitting an auth/network/unknown error while guessing sees an English toast — the most visible untranslated path, on the unhappy path everyone eventually hits. Fix: return i18n keys from getUserFriendlyErrorMessage and translate at the toast site.

13. Integration test seams the merge introduced are uncovered

QA #1, #2, #3 · evidence: game.service.submit-guess.test.ts:143 mocks findGuessMatchCandidates → []

Proximity wiring through submitGuess never actually fires in tests (empty candidate list), findGuessMatchCandidates tokenisation is wholly untested, partial×floor / partial×letter-penalty combos are untested, and there's no e2e for the partial badge or warmer banner. Fix: harness case returning a related candidate (assert hint + anti-leak), extract & unit-test the tokenizer, add the partial-combination cases, add a daily-game e2e for both UI elements.
