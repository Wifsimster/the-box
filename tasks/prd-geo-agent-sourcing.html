Geo Agent Sourcing — Implementation Plan

issue #331 · rev 1 · 2026-07-10 · grounded in code as of e9968db (v2.139.0)

TL;DR

Issue #331's direction is right: the pin is the bottleneck, the agent proposes, consensus disposes. This plan grounds it in the actual code and changes five things:

  1. Fixes an auto-promote hole — under today's consensus rules, 5 tightly-clustered agent pins would satisfy MIN_PINS_TO_PROMOTE and silently become ground truth with zero humans. Agent pins must never count toward the promote gate.
  2. Corrects the structured-coordinates assumption — MapGenie/Wand adapters only fetch a flat og:image today; no adapter extracts marker coordinates. Phase 4 is real new work, and the hard part is screenshot↔landmark association, not scraping.
  3. Drops geo.promote_pin from the agent surface — promotion already exists as the admin override route; a promote tool (even "admin-gated") violates propose/dispose for zero gain.
  4. Specs the missing provenance model — geo_pin_submission has no source column and requires a non-null user_id; agent pins need a migration, not just a flag.
  5. Separates key minting from streamer self-service — scopes exist but are unenforced and not settable at mint; geo-agent keys must be admin-minted only.

Ship order: Phase 1 (diagnostic endpoint + admin card) is standalone and unblocks manual pinning toward the 10-game GeoGamers gate today.

2Target architecture

Two deliberate simplifications vs. the issue:

  • HTTP first, MCP as a thin wrapper. The capability surface is a plain key-authenticated HTTP API (testable with curl, e2e-testable with Playwright's request fixture). The MCP server is a ~200-line stdio wrapper that maps tools 1:1 to endpoints — it holds no logic and no privileges of its own, so a compromised/hallucinating MCP layer can't do anything the key can't.
  • No geo.promote_pin tool. Admin promotion already exists (POST /api/admin/geo/candidates/:id/override, session-authed, audited, with a purpose-built review UI). Re-exposing it behind an API key — a bearer credential that can leak — only widens the blast radius.

3Design decisions

3.1 Identity of an agent pin

geo_pin_submission.user_id is NOT NULL with an FK to user, and uniqueness is (user_id, candidate_id). Options:

OptionVerdict
Synthetic "system user" per agent key Rejected — pollutes user-facing joins (leaderboards, contributor stats, Better Auth), and geo_contributor_stats / shadow-ban logic would need agent carve-outs everywhere.
Nullable user_id + agent_key_id FK + CHECK (exactly one owner) Chosen. One migration; human paths unchanged (their user_id stays NOT NULL in practice); agent pins are trivially filterable; revoking the key orphans nothing.

Contributor stats, tier promotion, token rewards, and shadow-ban (geo-reward.service.ts) apply only to pins with a user_id — agent pins earn nothing and can't be shadow-banned (they're governed by budgets + kill switch instead). applyConsensus's pinOwners map simply skips ownerless pins, which it already tolerates.

3.2 Consensus: weights sharpen, humans promote

Extend the pure service (geo-consensus.service.ts) — no infra change:

export type GeoPinSource = 'human' | 'agent_structured' | 'agent_vision'

export const GEO_CONSENSUS_SOURCE_WEIGHTS: Record<GeoPinSource, number> = {
  human: 1.0,
  agent_structured: 0.6,   // scraped marker coords, provenance attached
  agent_vision: 0.25,      // LLM localization, experimental
}

// effective weight = confidenceWeight * sourceWeight
// GeoPinLike gains: source?: GeoPinSource   (default 'human')

// Promote gate (v3):
//   humanAcceptedCount >= GEO_CONSENSUS_MIN_PINS_TO_PROMOTE (still 5)
//   && confidence >= 0.5
// Agent pins: contribute to centroid + sigma + confidence, NEVER to the count.

export const GEO_CONSENSUS_VERSION = 3  // bump (was 2)
Why this rule: it preserves the exact property the issue asks for ("machine writes can never directly mutate ground truth") as an invariant of the math, not a policy hope. Even 1,000 colluding agent pins cannot promote a candidate; they can only make the eventual human-driven consensus more precise — or get rejected as outliers by the existing 2σ filter. A property test asserts promote === false for any all-agent pin set (§8).

Note consensus_version is already persisted on geo_screenshot_meta, so v2/v3 metas stay distinguishable. Rejected-agent-pin ratios per key become a health metric (§9).

3.3 Key scopes & minting

// packages/types/src/index.ts — extend the union
export type ApiKeyScope =
  | 'read:public' | 'read:self' | 'stream:self' | 'webhooks:self'
  | 'geo-agent:read'      // health, needing-content, candidates
  | 'geo-agent:ingest'    // trigger existing ingest jobs
  | 'geo-agent:propose'   // submit downweighted pins
  • Minted only via a new admin route (POST /api/admin/agent-keys, adminMiddleware, audited) — never via streamer-keys.routes.ts. apiKeyRepository.create() already accepts a scopes override, so the repo needs no change.
  • Geo-agent scopes and streamer scopes are mutually exclusive on one key (validated at mint). A geo-agent key on the public streamer API gets INSUFFICIENT_SCOPE, and vice versa.
  • Revocation = existing soft-revoke (is_active=false) — instant, since findByHash filters on is_active.
  • Progressive trust: Phase 2 keys get read only; ingest and propose are granted per key as later phases ship.

4Schema — one migration (Phase 4)

// packages/backend/migrations/2026MMDD_geo_agent_pins.ts
alter table geo_pin_submission
  alter column user_id drop not null,
  add column source varchar(20) not null default 'human',
      -- 'human' | 'agent_structured' | 'agent_vision'
  add column agent_key_id integer null references api_keys(id),
  add column agent_rationale varchar(500) null,
  add column agent_model varchar(100) null,   -- e.g. model id for vision pins
  add column vision_pass smallint not null default 0,
  add constraint geo_pin_owner_check check (
    (user_id is not null and agent_key_id is null and source = 'human') or
    (user_id is null and agent_key_id is not null and source <> 'human')
  );

-- human uniqueness unchanged: (user_id, candidate_id)
create unique index geo_pin_agent_uniq
  on geo_pin_submission (agent_key_id, geo_screenshot_candidate_id, vision_pass)
  where agent_key_id is not null;
create index geo_pin_source_idx on geo_pin_submission (source)
  where source <> 'human';
  • geo_screenshot_meta.promoted_via stays consensus | admin — there is intentionally no 'agent' value.
  • Down migration deletes agent pins, then restores NOT NULL — stated explicitly in the file so a rollback is data-destructive by design, not by surprise.
  • GeoPinSubmissionRow (geo-pin.repository.ts:12) and geoPinRepository.submit() gain the new fields; listPendingByCandidate starts selecting source so the worker can feed it to evaluateConsensus.

5The agent API surface

New mount /api/agent/v1/geo (agent-geo.routes.ts) behind: requireApiKey() (reused) → requireScope(…) (new, wraps the existing hasScope helper) → budget middleware → Zod validation → audit. Envelope and error codes follow the public API (UNAUTHORIZED, INSUFFICIENT_SCOPE, RATE_LIMITED, VALIDATION_ERROR + new BUDGET_EXHAUSTED, AGENT_API_DISABLED).

Deliberately absent: promote/override, candidate reject/demote, map selection, user or PII reads, raw SQL, file access. Responses never include other users' identities — candidate payloads carry pin counts, not pin owners.

Reads are rate-limited but not audit-logged (volume). Writes always are. listRecent({ adminId: 'apikey:…' }) makes the existing admin audit view filterable per key with zero UI work.

6Phased delivery

Phase 1 — "one pin away" diagnostic ship now · S

No agent, no keys — immediately unblocks manual pinning toward the 10-game gate (prod is at 9/10).

  • GET /api/admin/geo/games-needing-content — games with an active map and ≥1 non-promoted candidate, no eligible meta; per game: { gameId, name, candidateCount, topPinCount, pinsToNextThreshold, bestCandidateId }, sorted by topPinCount desc (closest to the next GEO_CONSENSUS_THRESHOLDS recompute first). pinsToNextThreshold = nextThreshold(topPinCount) − topPinCount.
  • Admin card next to GeoGamersHealthCard in the Géo tab: top-10 list, deep-links into GeoReviewQueue (admin can promote via the existing override) and to /geo/contribute.
  • Also surface gamesOnCooldown context: a game can be "eligible-blocked" purely by challenge cooldown — distinguish that from "needs pins".

Phase 3 — ingest trigger S

  • POST /games/:gameId/ingest behind geo-agent:ingest + Redis daily budget. Pure enqueue — reuses tick logic, circuit breakers, geo_source_config, dedup, and license/attribution capture untouched.
  • Source allowlist = RunnableTier ∩ geo_source_config.is_enabled; anything else → VALIDATION_ERROR. The agent cannot add sources.
  • Response includes geo_ingest_attempt correlation ids so the agent can poll outcomes via the read surface (add attempts to the candidates/health reads).

Files: agent-geo.routes.ts · geo-ingest-tick-logic.ts (export reuse only) · infrastructure/redis budget helper · MCP tool geo_ingest_game

Phase 4 — pin proposals, structured tier L · design-gated

  • Migration §4 + consensus v3 (§3.2) + repo/worker plumbing (submit, listPendingByCandidate → pass source).
  • POST /candidates/:id/pins behind geo-agent:propose + hourly budget. rationale required (≤500 chars) — it is the review artifact.
  • Geo Review UI: source filter on GeoReviewQueue; agent pins rendered in a distinct color on the GeoReviewPanel map with rationale/model tooltip; count badge "N agent pins" on candidate rows.
  • Structured-coordinate extraction is scoped as its own worker (maps-fetch-mapgenie phase-2 crawler slot): crawl marker JSON, normalize provider coords → the map's [0,1] space (per-map affine calibration, stored alongside the map row), and — the hard part — match screenshots to landmarks. First target: sources where the screenshot itself carries location context (wiki location-page images), not generic landmark lists. If association confidence is low, the agent should not pin — precision over recall; a wrong structured pin at weight 0.6 poisons a centroid.

Files: migrations/2026MMDD_geo_agent_pins.ts · geo-consensus.service.ts (+tests) · geo-pin.repository.ts · geo-consensus-logic.ts · geo-reward.service.ts (skip ownerless) · agent-geo.routes.ts · GeoReviewQueue.tsx / GeoReviewPanel.tsx · docs/geo-mode.md

Phase 5 — vision proposer M · accuracy-gated

  • Study before write access: offline harness packages/backend/scripts/geo-vision-eval.ts — sample K≥50 promoted metas (known ground truth), run vision localization (screenshot + map image), report median normalized distance, % within consensus_radius (default 0.03), and % within 0.1. Enable bar: ≥40% within radius and median < 0.1; below that, vision pins add noise even at weight 0.25 — skip the tier, keep structured-only.
  • If passed: source='agent_vision', up to 3 independent passes per candidate (vision_pass 0..2, different prompt seeds), each a separate 0.25-weight voter so the 2σ filter can discard disagreeing passes.
  • Track per-key rejected-ratio; auto-pause proposals from a key whose 7-day rejection ratio exceeds the human shadow-ban bar (0.6) — same philosophy, budget-level enforcement.

Files: scripts/geo-vision-eval.ts · agent-geo.routes.ts (allow source=vision) · budget/pause helper · docs/geo-agent-api.md (published accuracy numbers)

Phase 6 — backfill discovery worker M

  • New GeoJobData kind backfill-tick: rank games by "distance to eligibility" (has map? has candidates? top pin count?) using the Phase-1 query, then enqueue ingest for the top N sub-threshold games — inverting today's tick, which tops up already-resolved games to 30 candidates.
  • Steady-state automation of what the external agent does exploratively; no LLM in-process — it reuses the same ranked query + ingest path. Cron-registered like other geo jobs, off by default (GEO_BACKFILL_ENABLED).

Files: queues.ts (kind) · workers/geo-backfill-logic.ts · geo.worker.ts router · env.ts

8Test plan

  • Unit (node test runner): consensus v3 — property test: any pin set with <5 accepted human pins ⇒ promote=false regardless of agent pins; weight composition (confidence × source); 2σ rejection of a deliberately wrong agent pin; version bump. Budget counter math (UTC day/hour rollover). Scope middleware matrix (each endpoint × each scope × disabled flag).
  • Route tests: mint/revoke agent keys admin-only; streamer key rejected on agent routes with INSUFFICIENT_SCOPE; propose-pin path increments pin_count and enqueues evaluate-consensus only at thresholds; no rewards/contributor mutation for agent pins.
  • E2E (Playwright): Phase 1 card renders + deep-links (extend geo-admin.spec.ts); Geo Review shows/filters agent pins; admin key panel mint flow (plaintext shown once).
  • Offline: vision eval harness output committed to the PR that proposes enabling Phase 5 — the numbers are the review artifact.

9Metrics & rollout

  • North star: eligibleGames (already on the health endpoint) — the 10-game gate is the whole point.
  • Per key: proposals, accepted/rejected ratio, promoted-with-agent-participation count (join meta → candidate → agent pins), budget exhaustions — derivable from audit log + pin table; add to the admin health card in Phase 4.
  • Rollout: each phase ships dark behind its flag; Phase 4+ enables on one hand-picked game first; revert path = kill switch + (worst case) DELETE FROM geo_pin_submission WHERE source <> 'human' — ground truth is untouchable by construction, so recovery never involves geo_screenshot_meta.

10Open questions from #331 — resolved

QuestionResolution
Where does the agent run? Both, sequenced: external (Claude Code + MCP key) for exploration in Phases 2–5; the in-app Phase-6 worker takes over steady-state backfill with no LLM in-process. The HTTP surface is identical for both, so nothing is rebuilt.
Is vision good enough even as a downweighted voter? Empirical gate with published numbers (§ Phase 5): ≥40% within consensus radius and median < 0.1 on ≥50 known metas, else the tier stays off. Not a judgment call.
Ever auto-promote high-confidence structured pins? No — removed from the design. Promotion is human-count-gated in the math (§3.2) or explicit admin action. If structured pins prove extremely accurate, the cheap lever is raising their weight (sharper centroids → faster human-driven promotion), never bypassing the gate.
