Mobile Layout & Navigation — Personas Meeting Internal

branch: claude/mobile-layout-improvements-2XZOp · date: 2026-05-17 · audience: engineering

Four personas were spawned in parallel to review “The Box”’s application layout and navigation, with a brief to improve the mobile-first experience. Each had a narrow lens — UX/product, real-device QA, frontend implementation, and WCAG 2.2 accessibility — and reported back with prioritised, file-level findings. This note captures the synthesis, the agreed decisions, and what shipped.

Verdict

Adopt a persistent mobile bottom navigation bar. All four personas converged independently: a hamburger drawer is the wrong primary navigation for a daily-engagement game. It buries every destination behind a three-tap interaction in the hardest-to-reach corner of the screen, exposes no information architecture, and offers no “you are here” signal. The drawer is kept as a complete overflow/account menu; the bottom bar becomes the one-tap primary nav.

Starting point

No sidebar exists. Mobile navigation is a single left-edge hamburger Sheet drawer; desktop is a top horizontal nav plus a user dropdown. The Header.tsx file is 419 lines and duplicates the navigation and account sections between the two layouts. Nav items render as Button size="sm" (32 px tall). There is no active-route indication, no skip link, and no env(safe-area-inset-*) handling on the header or the drawer.

Personas & Findings

UX / Product Designer Re-architect

mobile-first · daily-habit loop

A hamburger fits content-heavy sites with deep, infrequent navigation — “The Box” is the opposite: a small destination set, visited daily, on phones. The drawer adds friction exactly where an engagement app must remove it, and opens from the top-left, the worst one-handed reach.

  • P0 Persistent bottom bar — Home · Play · Leaderboard · Profile — with a visible active indicator and ≥48 px targets; hidden on /play and /geo.
  • P1 Surface the daily-reward badge in the header — it was buried inside the drawer, invisible on first paint, undermining the streak loop.
  • P2 Decide Geo’s status; bump remaining 32 px controls to 44 px.

Mobile QA / Real-Device Safe-area risks

notch · keyboard · dvh · PWA standalone

A fixed bottom bar collides with the iOS home indicator and Android gesture bar unless it reserves env(safe-area-inset-bottom). The sticky header has no safe-area-inset-top, so in installed-PWA mode (black-translucent status bar) it renders behind the notch. The left Sheet ignores safe areas entirely.

  • P0 Header padding-top = inset-top; bottom bar = inset-bottom; Sheet left/right insets; bottom bar must hide while the virtual keyboard is open on form pages.
  • P1 Shell heights 100vh → 100dvh; reserve content/footer space so nothing scrolls under the fixed bar.
  • P2 Enable a Mobile WebKit Playwright project — iOS is where these bugs live.

Frontend Engineer Build plan

React 19 · Tailwind v4 · shadcn/ui

Hand-roll a small BottomNav on NavLink — do not abuse Radix Tabs (its panels/roving-tabindex fight react-router) and do not install the shadcn sidebar block (heavy, solves a problem a 5-link top bar doesn’t have). The 419-line Header duplicates its nav and account lists; extract one typed config consumed by all three surfaces.

  • P0 Shared nav-items.ts config; Header refactor; NavLink for active state (Home needs end against the localized prefix).
  • P1 BottomNav component mounted in LanguageLayout; z-index re-stack so PWA banners float above the bar.
  • P2 A --bottom-nav-h token so the bar height and reserved padding can’t drift.

Accessibility (WCAG 2.2 AA) 3 AA gaps

landmarks · targets · focus

Three concrete AA gaps, none needing architectural change: no skip link (SC 2.4.1), 32 px nav targets with sub-24 px spacing (SC 2.5.8), and no active-route signalling at all — no aria-current (SC 4.1.2 / 2.4.8).

  • P0 Skip link to <main>; ≥44 px targets; aria-current="page" + a non-colour active indicator (SC 1.4.1 / 1.4.11).
  • P1 Distinct aria-label on each <nav> landmark; drawer becomes a real <nav>; scroll-padding for Focus-Not-Obscured (SC 2.4.11).
  • P2 aria-hidden on decorative icons.

Decisions

Scope — full consensus set. Bottom nav + Header dedup refactor + all P0/P1 accessibility and safe-area fixes. (Confirmed with the requester.)
Bottom-bar destinations — Home · Play · Leaderboard · Profile. Four items for generous thumb spacing. Geo (alpha) stays reachable from the Home page and the drawer rather than taking a top-level tab. (Confirmed with the requester.)

Out of scope — follow-ups

  • Geo escape affordance. The fullscreen /geo route has no visible exit control (UX flagged P0). It is a route-specific concern, separate from this layout/nav pass — track separately.
  • Mobile E2E. Enable a Mobile WebKit Playwright project and add specs for bottom-nav navigation, safe-area insets, and the keyboard-open state.
  • PageHero reduced motion. Its Framer Motion entrance animations are JS-driven and not covered by the global CSS prefers-reduced-motion rule (a11y P2).
  • Landscape. Consider auto-hiding the bottom bar in landscape phone orientation, where header + bar eat ~30% of the viewport (QA consideration).
