Architecture

Stack Node 24 · Express 5 · Knex/Kysely · PostgreSQL 16 · Socket.io 4 · BullMQ + Redis · React 19 · Vite 7 · TypeScript

The backend is a pragmatic three-layer clean architecture with strict unidirectional dependency: Presentation → Domain → Infrastructure. The domain layer never imports from infrastructure — that's the rule the diagram below enforces visually. Companion reference: docs/architecture.md (français).

1. Request lifecycle

Press Play trace to follow a real request — a player submitting a guess on the daily challenge. Hover any node to highlight it. The red strikethrough arrow at the bottom is the forbidden reverse import direction.

POST /api/guesses → auth → validate → game.service → fuzzy-match → leaderboard → repo → postgres + socket broadcast
Presentation Domain Infrastructure request flow forbidden direction
The rule. Domain services may import from @the-box/types only. They never import from infrastructure/. Repositories, Better Auth, Socket.io, BullMQ — all infrastructure. They are wired into services via dependency injection at app boot, not via direct imports inside the service modules.

3. Presentation layer

packages/backend/src/presentation/ — Express controllers and middleware. Controllers are thin: they parse, authenticate, validate, then delegate to a domain service.

  • Routes — routes/{achievement,admin,auth,daily-login,game,leaderboard,user}.routes.ts. All mounted under /api/.
  • Middleware — auth.ts (Better Auth session check), validation.ts (Zod), request logging.

Realtime (Socket.io)

Two room types:

  • challenge_<id> — dynamic per-challenge room. Clients join when entering a leaderboard view; server emits leaderboard_update when a guess lands.
  • admin — singleton room for admins watching job progress (job_progress, job_complete, job_failed).

Detailed event reference: docs/realtime.md.

7. Structural decisions

  1. Single shared types package — one source of truth for cross-package interfaces.
  2. Repository pattern — DB access abstracted behind repos so domain services stay testable.
  3. Thin controllers — routes only handle HTTP concerns.
  4. Zustand over Redux — smaller API, less boilerplate, persist middleware where needed.
  5. BullMQ + Redis — durable retry semantics for imports and emails.
  6. Fuzzy matching — tolerant guess matching is core to UX, lives in a dedicated domain service.
  7. Single Docker image in production — Node serves the built SPA and the API on port 80.

8. How to extend safely

  1. New endpoint? Add a route in presentation/routes/, validate with Zod, call a domain service.
  2. New business rule? Put it in a domain service. If it needs data, inject a repository — do not import infrastructure directly.
  3. New table? Create a migration under packages/backend/migrations/ (date-prefixed YYYYMMDD_name.ts). Migrations run automatically via docker-entrypoint.sh.
  4. New background job? Define a queue in infrastructure/queue/queues.ts and a worker in infrastructure/queue/workers/. Wire it from src/index.ts.
  5. New realtime event? Document it in docs/realtime.md and emit from the appropriate domain hook.
